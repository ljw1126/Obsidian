

```text
개요
주석 규칙을 정의함

아래 규칙 예시에 참조하여 사용

규칙
형식

한명일 경우: {종류}:[담당자] 내용

여러명일 경우: {종류}:[담당자1][담당자2] 내용


예시

1. TODO (할 일)
목적: 해야 할 작업을 표시. 해당 부분을 구현 OR 수정하지 않으면 기능 동작에 문제가 있습니다.
예시: // TODO:[a][b] 이 함수에 오류 처리 구현하기

2. FIXME (수정 필요)
목적: 수정 OR 개선이 필요한 코드지만 기능 동작에는 문제가 없음을 표시합니다.
예시: // FIXME:[a] 이 알고리즘은 비효율적이므로 최적화가 필요함

3. NOTE (참고)
목적: 코드에 대한 추가 정보나 설명을 제공합니다.
예시: // NOTE:[a] 이 함수는 입력값이 항상 양수라고 가정함

4. HACK (임시방편)
목적: 임시 해결책이나 이상적이지 않은 솔루션을 표시합니다.
예시: // HACK:[a] 이 모듈을 리팩토링할 때까지의 임시 해결책

5. BUG (버그)
목적: 버그가 있지만 현재 수정할 수 없는 상황 또는 정책/수동으로 처리하는게 이득인 상황에서 인지하기 위한 표시
예시: // BUG:[a] 입력값이 0일 때 함수가 실패함

6. OPTIMIZE (최적화)
목적: 성능 개선이 필요한 코드를 제안합니다.
예시: // OPTIMIZE:[a] 이 루프의 효율성을 높일 수 있음

7. XXX (주의 필요)
목적: 주의가 필요하거나 문제가 있는 코드를 표시합니다.
예시: // XXX:[a] 이 섹션은 완전한 재작성이 필요함

8. REVIEW (검토)
목적: 팀원들의 검토가 필요한 코드를 표시합니다.
예시: // REVIEW:[a] 이것이 동시성 처리를 위한 최선의 접근법인가?

9. IDEA (아이디어)
목적: 잠재적인 개선사항이나 기능을 제안합니다.
예시: // IDEA:[a] 여기에 캐싱을 추가하여 성능을 향상시킬 수 있음

10. QUESTION (질문)
목적: 코드나 구현에 대한 질문을 제기합니다.
예시: // QUESTION:[a] 이 엣지 케이스를 다르게 처리해야 할까?

11. TEMP (임시)
목적: 테스트 OR 특수한 상황에서 임시적인 기간 OR 상황 동안에만 작동하는 코드임을 표시
예시: // TEMP:[a] 기능 A가 배포된 이후에는 코드 제거 필요함


region 쓸 때도 위 규칙 그대로 적용가능
// region TEMP:[a] 기능 A가 배포된 이후에는 코드 제거 필요함

// endregion
```
