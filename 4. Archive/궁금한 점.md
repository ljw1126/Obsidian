  싶습니다.


  결론부터 말씀드리면, AI 시대일수록 리팩터링 능력은 '코딩 실력'을 넘어 '의사결정 능력'이자 '품질 관리 능력'이 됩니다.


  1. "무엇을 시킬 것인가"를 결정하는 것은 결국 사람입니다.
  AI는 "이 코드를 리팩터링해줘"라고 하면 수많은 옵션을 제시할 수 있습니다. 하지만:
   * 이 상황에서 '함수 추출하기'가 최선인가, 아니면 '다형성으로 교체하기'가 최선인가?
   * 현재 우리 팀의 컨벤션과 비즈니스 복잡도에 비추어 볼 때 이 설계가 과한(Over-engineering) 것은 아닌가?
  이러한 전략적 판단은 AI가 아니라, '보는 눈'을 가진 개발자가 내리는 것입니다. 리팩터링 도서를 공부하는 것은 AI에게 내릴 '정교한 명령서'를 작성하는 법을 배우는 것과 같습니다.


  2. AI의 결과물을 '검증'할 수 있어야 합니다.
  AI도 가끔 그럴듯해 보이지만 유지보수가 불가능한 코드를 만들거나, 미묘한 버그(Hallucination)를 심어놓기도 합니다.
   * 테스트 코드가 왜 중요한지, 어떤 단위로 쪼개야 안전한지 모른다면 AI가 만든 코드가 '맞아 보이는' 것인지 '정말 맞는' 것인지 확신할 수 없습니다.
   * "직접 쳐보는 경험"은 코드의 무게감을 느끼게 해줍니다. 내가 직접 고생하며 고쳐본 코드는 나중에 AI가 만든 코드의 결함을 잡아내는 강력한 기준점이 됩니다.


  3. "보는 눈"은 근육과 같습니다.
  수학 문제를 풀 때 해설지를 보면 다 이해되는 것 같지만, 직접 풀면 막히는 것과 비슷합니다.
   * 눈으로만 보면 '지식'에 머물지만, 직접 타이핑하며 테스트 통과 여부에 안도하고 에러에 당황해보는 과정은 '경험'이 됩니다.
   * 이 경험이 쌓여야만 코드를 보자마자 "아, 여기는 나중에 문제가 생기겠는데?"라는 직관이 생깁니다. AI는 이 직관을 보조할 뿐, 대신 만들어주지는 못합니다.


  💡 추천드리는 학습 방향
  1년 전처럼 모든 설정을 직접 하느라 진을 뺄 필요는 이제 없습니다. (그건 제가 해드릴 수 있으니까요!) 대신 다음 단계에 집중해 보세요.


   1. 설계의 의도 파악: 제가 만들어드린 초기 코드에서 책의 각 단계(함수 추출, 변수 인라인화 등)를 진행할 때, "왜 마틴 파울러는 여기서 이 방법을 썼을까?"를 끊임없이 질문해 보세요.
   2. AI와 토론하기: 직접 리팩터링을 해보시다가 막히거나, 혹은 더 나은 방법이 궁금할 때 저에게 물어보세요. "나는 여기서 함수를 추출하려고 하는데, 너는 어떻게 생각해?"라고 묻는 과정
      자체가 엄청난 공부가 됩니다.
   3. 테스트의 안정감 느끼기: 리팩터링을 한 줄 하고 테스트를 돌릴 때의 그 '안도감'을 체득해 보세요. 그 안도감이 결국 좋은 설계를 향한 동기부여가 됩니다.

   1. "함께 삽질할 동료"를 구하세요

         사람들이 모입니다.


   2. 구체적인 성과물 제시
       * "매주 블로그 포스팅 1개" 혹은 "나만의 리팩터링 카탈로그(GitHub) 만들기"처럼 눈에 보이는 목표를 제시하면 참여율이 높아집니다.


   3. 'AI + TDD'라는 차별점 강조
       * 단순히 책만 읽는 스터디는 많습니다. "AI에게 리팩터링 가이드를 요청하는 법(Prompt Engineering)"과 "안전한 수정을 보장하는 TDD"를 같이 연습한다는 점을 강조하면 주니어부터
         시니어까지 관심을 가질 만한 주제가 됩니다.

  🛠 추천하는 스터디 방식 (AI 활용 workflow)


   * 준비: 챕터별로 AI(Gemini, ChatGPT 등)에게 "이 리팩터링 기법이 필요한 '안 좋은 예제 코드'를 TypeScript로 짜줘"라고 요청합니다.
   * 실전:
       1. AI가 만든 안 좋은 코드에 대해 테스트 코드를 먼저 작성합니다 (TDD).
       2. 책의 절차에 따라 직접 리팩터링합니다.
       3. 중간에 막히면 AI에게 "이 단계에서 책에서는 이런 절차를 권하는데, 내 코드에 어떻게 적용할까?"라고 물어봅니다.
   * 공유: "나는 AI가 알려준 방식보다 책의 방식이 더 이해하기 쉬웠다" 혹은 "AI가 제안한 이 변수명이 훨씬 직관적이었다" 등의 경험을 나눕니다.

private StatementData createStatementData(Invoice invoice) {
    // performances를 먼저 만들고
    List<EnrichPerformance> enrichedPerformances = invoice.performances().stream()
            .map(perf -> new EnrichPerformance(perf, playFor(perf), amountFor(perf), volumeCreditsFor(perf)))
            .toList();

    // 완성된 리스트를 바탕으로 합계를 구해 생성자로 한 번에 넘김
    return new StatementData(
            invoice.customer(),
            enrichedPerformances,
            totalAmount(enrichedPerformances), // data 대신 리스트를 인자로
            totalVolumeCredits(enrichedPerformances)
    );
}
