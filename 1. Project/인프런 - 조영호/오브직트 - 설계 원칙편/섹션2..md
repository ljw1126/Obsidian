

**회귀(regration) 버그** 
리팩터링 후 정상적으로 동작하던 기능에서 문제가 생긴경우 

**회귀(regration) 테스트**
리팩터링 후  정상적으로 동작하던 기능에서 문제가 없는지 검증하는 테스트


>[!info]
>리팩터링을 하기 전에 먼저 자체 테스트 코드를 만들어야 한다. 테스트를 자주 실행시켜라. 컴파일할 때마다 테스트를 하고, 적어도 하루에 한번 모든 테스트를 실행시켜라. - 마틴 파울러, 리팩터링

> 테스트하기 쉽게 코드를 설계해라 (테스트 가능하게 설계하라)
> 하지만 말처럼 쉽지 않다. 
- 테스트 관점에서 객체를 바라봐야한다
	- 잘 설계된 객체는 상태를 내부에 감추고, 외부에서는 인터페이스를 통해 접근 가능하도록 허용
	- 메시지를 수신한 객체는 스스로 상태를 변경
	- 내부상태에 직접 접근하도록 하면 안된다

> 상태 기반 검증 테스트

> 상호작용 기반 검증 방식의 테스트 
> - mock의 verify 같은 느낌도 든다


`Game`의 경우 키보드를 통해 입력을 받아 상태를 변경한다. 
- 입력 : 특정한 키보드 입력에 대해 
- 출력 : 특정한 결과가 콘솔에 출력되는지 검증 한다

```java
Scanner // Game 내부에서 생성되는 Scanner 인스턴스

System.out.println("..") // static 멤버 out을 통해 결과 출력
```
- run() 메서드 내부에 선언되어 있다보니 제어가 어렵다.. 
	- = 숨겨진 의존성 


좋은 방법은 아니지만 키보드 입력과 출력 인터셉트
- 키보드 입력을 설정하고 콘솔 출력 결과를 확인하기 위해 System 클래스 내부 상태 변경

```java
// 화면 출력 결과를 output에 누적
OutputStream output = new ByteArrayOutputStream();
System.setOut(new PrintStream(output));

// 키보드를 통해 go north와 quit 명령어 입력 
System.setIn(new ByteArrayInputStream("go north\nquit\n".getBytes()));
```

`System.setOut(..)`, `System.setIn(..)`을 테스트 마다 사용하면 상태 공유가 되서 테스트가 실패할 수 있다.

> 테스트는 리팩터링 실패에 대한 알람인 동시에
> 더 좋은 설계로 리팩터링할 위치를 알려주는 가이드

> 테스트하기 쉬운 코드/클래스는 좋은 설계일 확률이 높음 

> 테스트하기 어려운 클래스는 설계가 좋지 않다

> 테스트하기 어려운 코드를 만났다면 코드를 개선하라는 시그널로 받아 들여야 한다.

**✅ 설계 원칙은** 
- 코드의 어떤 문제가 있는지 평가할 수 있는 기준을 제공
- 어떤 목표로 리팩터링하면 되는지 가이드를 제공
