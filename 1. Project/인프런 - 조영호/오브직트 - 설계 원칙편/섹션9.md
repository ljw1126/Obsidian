### 9-2. 인터페이스 분리 원칙
> 날짜 관련 예제 Schedule 사용
- DateInterval
	- TemporalFilter 인터페이스
	- 삭제 후 RecurringPlan에 apply 추상 메서드 추가
	- 인터페이스 변경이 사용하지 않는 MonthlyPlan 구현체한테 까지 영향을 끼쳐서 변경에 취약한 설계 듦
		- WeeklyPlan에서만 인터페이스 구현하여 사용하도록 한다ㄴ

**인터페이스 오염(Interface Pollution)**
- 인터페이스에 사용되지 않는 불필요한 오퍼레이션 포함
- 이는 곧 LSP 위반으로 이어진다
	- 변경에 취약한 설계가 들어난다 (= 설계가 취약하다)

> 인터페이스 수정의 원인은 클라이언트라는 사실이 핵심
- `RecurringPlan` 인터페이스를 DateInterval과 Schedule 클래스에서 의존하고 있다
	- 두 클래스에서 인터페이스가 변경된다 
	- 변경될 이유가 여러 개이다

> 해결 방법은 클라이언트 별 인터페이스 분리

>[!info] 인터페이스 분리 원칙 (Interface segregation Principle, ISP)
>**클라이언트가 자신이 사용하지 않는 메소드에 의존하도록 강제되어서는 안 된다.**
>
>어떤 클라이언트가 자신은 사용하지 않지만 다른 클래스가 사용하는 메서드를 포함하는 클래스에 의존할때, 그 클라이언트는 다른 클라이언트가 그 클래스에 가하는 변경에 영향을 받게 된다. 가능하면 이런 결합을 막고 싶다. 따라서 인터페이스를 분리하기를 원한다 - 로버트 C 마틴, 애자일 소프트웨어 개발


> 인터페이스를 여러 개를 하위 모듈에서 구현할 수 있구나 

**의존성 역전 원칙 만족**
- 상위 수준 모듈과 하위 수준 모듈 모두 추상화에 의존

**LSP 만족**
- 클라이언트 관점에서 모든 클래스는 인터페이스를 치환 가능

**개방-폐쇄 원칙 만족**
- 기존 코드 수정 없이 새로운 클래스 추가만으로 기능 확장 가능

**인터페이스는 클라이언트 소유**
> 클라이언트와 인터페이스는 응집도 측면에서 같은 패키지에 있어야 한다

**패키지 수준의 단일 책임 원칙**
- 함께 변경되는 클라이언트와 인터페이스가 같은 패키지에 포함되야 한다✅

**패키지 사이의 의존성 역전 원칙**


**📌 인터페이스 분리 원칙 요약**
- 각 클라이언트가 인터페이스/클래스의 일부 메서드에만 의존한다면
- 변경 시에 영향을 통제할 수 있도록 클라이언트에 따라 인터페이스를 분리하라 
	- 구현 클래스는 하나이더라도, 인터페이스 분리를 통해 노출을 지역화 할 수 있구나 
	- 이를 통해 클래스가 불필요한 이유로 변경되는 것을 줄일 수 있다.



### 9-3. 인터페이스 분리하기 
> 다시 Game 예제로 돌아와 ISP 원칙 적용

**인터페이스 분리 원칙은 인터페이스 관점에서의 단일 책임 원칙**
> 인터페이스를 변경하도록 만드는 클라이언트를 기준으로 인터페이스를 분리하라
- 변경의 이유를 하나만 가지는 인터페이스를 설계할 수 있습니다.

```text
Source <interface>
- find(name)
- remove(item)

Target <interface>
- add(item)
  
Carrier // 위에 두 인터페이스를 상속
- List<Item> items();


Source <- Carrier -> Target
```

**순서** 
- 인터페이스 세분화 
	- Carrier 가 가진 행위를 Target, Source 타입으로 분리 후 상속
- 아이템 이동 리팩터링 
	- Transfer 클래스의 변수 타입 수정 
		- 변수의 역할이 타입을 통해 명확히 알아볼 수 있도록 한다
- 아이템 파괴하기 리팩터링 
	- Destory 클래스의 변수 타입 수정
- 아이템 출력 로직 리팩터링 
	- Game 클래스가 Carrier 인터페이스만 의존하도록
- 아이템 던지기 리팩터링 
	- WorldMap이 Carrier대신 Target만 의존하도록 한다 
		- 불필요한 메서드 오버라이딩 안해도 됨

> 리팩터링 후 설계 평가 

>✅  ISP 원칙은 인터페이스 관점에서 단일 책임 원칙을 준수할 수 있도록 하는 가이드다

```text
추가된 테스트 명세 (GameTest.java):

   * look_command: 현재 방의 정보와 아이템 목록이 표시되는지 확인.
   * help_command: 도움말 메시지가 표시되는지 확인.
   * inventory_command: 인벤토리 목록이 올바른 라벨(인벤토리 목록: [...])로
     표시되는지 확인.
   * take_item_success/failure: 방의 아이템을 인벤토리로 가져오는지 확인.
   * drop_item_success/failure: 인벤토리의 아이템을 방에 버리는지 확인.
   * destroy_item_success/failure: 아이템 파괴 기능 확인.
   * throw_item_success/failure: 아이템을 월드맵(랜덤 위치)으로 던지는 기능
     확인.
   * unknown_command: 잘못된 명령어 입력 시 안내 메시지 출력 확인.
```

   - 하지만 현재 CommandParser는 그런 복잡한 상호작용 없이 단순히 Unknown 명령어를 반환하고 있으므로, 지금 당장은 전혀 필요 없는 코드입니다.

  💡 결론 및 가이드
  현재 상태에서는 InputOutput 의존성을 과감히 제거하는 것이 객체지향 원칙(SRP, ISP)에 더 부합합니다. CommandParser는 입출력 매체(콘솔, 스윙 GUI, 네트워크 등)에 의존하지 않을수록 더
  유연하고 테스트하기 쉬운 클래스가 됩니다.

### 9-6. 중복 코드 제거하기 (DRY 원칙)
> Game, GameLoop 인터페이스 를 신규 생성해서 중복을 중이고 하위 모듈과 협력 가능하도록 한다

**CLI, GUI 게임의 중복 코드를 제거하기 위한 전단계**
> 두 코드가 유사해지도록 리팩터링

```text
멤머 변수 : World, CommandParser 

메서드:
welcome(), play(), commandParser.parse(..) 호출 후 executeCommand(..)
```
- Game 클래스를 신규로 생성해서 합성을 통해 사용하도록 한다.
- 중복 코드를 제거하기 위한 전 단계 
	- 두 코드가 유사해지도록 리팩터링

 stop(), play() 메서드가 서로 상이하다 
 - 하위 수준 모듈 : CliGame, GuiGame 
 - ✅ 상위 수준 모듈 : Game
 - 상위 수준 모듈이 다양한 하위 수준 모듈과 협력할 수 있도록 해주는 설계 원칙이 있다.
 - `DIP 원칙을 적용한다`
	 - `GameLoop` 인터페이스를 추가하고, Game이 의존하도록 한다 
- GuiGame에 input()이 빈 메서드를 가진다 (ISP, LSP 위반)
- Input, Output 인터페이스를 분리한다.
	- Console은 둘다 구현하고 
	- GuiGame은 Output만 구현한다 
	- 이를 통해 LSP를 

✅ **유연하고 재사용 가능한 설계의 핵심**
> 요구사항이 변경되거나 추가될 때마다 설계 원칙에 따라 코드를 리팩터링한다

> 👷설계 원칙은 현재의 설계를 개선하기 위한 지침으로도 활용

>[!info] 설계 원칙
>- 책임 주도 설계의 품질 기준 제공
>- 리팩터링의 목표 제시

>[!info]
>Game이 상위 모듈이고 GuiGame, CliGame 하위 모듈과 인터페이스 기반 DIP 원칙 적용된다고 생각함
>이때 상위 모듈이 합성을 통해 하위 모듈을 주입받아 사용한다고 생각했지만, 그게 아니란다.

```text
> GuiGame과 CliGame 둘다 하위 모듈이고 상위 모듈이 Game 이라고 이해했습니다. 인터페이스를 통해 합성을 한다는데 지금 하위 모듈에서 상위 모듈을 주입받아 사용하는 형태라 혼란스럽습니다.
   보통 DIP를 하게 되면 상위 모듈이 합성을 통해 하위 모듈 주입받아 사용한다고 생각했는데 혼란스럽습니다.
▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀▀
✦ 사용자님의 혼란은 "무엇이 상위 수준(High-level)이고 무엇이 하위 수준(Low-level)인가"에 대한 정의가 현재 코드 구조와 충돌하기 때문에 발생하는 당연한 의문입니다.

  DIP(의존성 역전 원칙)의 핵심은 "상위 수준의 모듈이 하위 수준의 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다"는 것입니다.

  현재 구조를 DIP 관점에서 다시 정의해 보겠습니다.

  1. 상위 모듈 vs 하위 모듈의 구분

  보통 우리가 "게임"이라고 부를 때, 진짜 중요한 비즈니스 로직(상위 수준)은 무엇일까요?
   - 상위 수준 (Policy/Logic): "플레이어가 북쪽으로 이동하면 방이 바뀐다", "아이템을 주우면 인벤토리에 들어간다"와 같은 게임의 규칙입니다. (현재 코드의 World, Command, Direction 등)
   - 하위 수준 (Detail/Mechanism): "콘솔에서 글자를 입력받는다", "스윙 버튼을 클릭한다"와 같은 입출력 방식입니다. (CuiGame, GuiGame, InputOutput 구현체들)

  2. 현재 구조의 문제점 (혼란의 원인)

  현재 CuiGame과 GuiGame 클래스를 보면, 이들이 전체 실행 흐름(Workflow)을 제어하고 있습니다.

    1 // CuiGame.java (현재 구조)
    2 public class CuiGame {
    3     private World world; // 상위 모듈(비즈니스 로직)을 주입받음
    4     // ...
    5     public void run() {
    6         while (isRunning()) {
    7             String input = io.input(); // 하위 모듈(IO) 사용
    8             Command command = commandParser.parseCommand(input);
    9             executeCommand(command); // 비즈니스 로직 실행
   10         }
   11     }
   12 }

  여기서 혼란이 생기는 이유는 "입출력이라는 하위 상세(Detail)"를 담당하는 CuiGame/GuiGame이, "게임 비즈니스 로직"인 World를 소유하고 제어하고 있기 때문입니다. 즉, 껍데기(UI)가
  알맹이(Logic)를 주입받아 사용하고 있는 모습이죠.

  3. DIP를 적용한 이상적인 구조 (역전의 모습)

  진정한 DIP는 게임의 흐름을 제어하는 상위 모듈이 따로 있고, 그 모듈이 InputOutput이라는 추상화(인터페이스)를 통해 하위 모듈(CUI/GUI)을 사용하는 형태여야 합니다.

  사용자님이 기대하신 모습(DIP):
   1. Game (상위 모듈): 게임의 전체 루프와 로직을 담당합니다. 구체적인 CUI/GUI는 모릅니다. 단지 InputOutput 인터페이스만 압니다.
   2. InputOutput (추상화): 입력을 받고 출력을 내보내는 규격입니다.
   3. ConsoleIO / GuiIO (하위 모듈): InputOutput을 실제로 구현합니다.

  이렇게 되면 Game이 ConsoleIO나 GuiIO를 주입받아 사용하게 되므로, 사용자님이 생각하신 "상위 모듈이 합성을 통해 하위 모듈을 주입받는 형태"가 됩니다.

  4. 현재 코드의 현실적인 타협안

  현재 코드는 CuiGame과 GuiGame 자체가 각 환경(콘솔/스윙)에 특화된 "실행 엔진" 역할을 하고 있습니다.

   - 만약 CuiGame을 "콘솔 환경에서의 게임 실행기"라고 본다면, 이 녀석은 하위 수준의 모듈이 맞습니다.
   - 이 하위 수준 모듈이 World(상위 로직)를 주입받아 사용하는 것은 "UI가 비즈니스 로직에 의존"하는 자연스러운 방향입니다. (DIP 위반이 아닙니다. 저수준이 고수준에 의존하는 것은
     정상입니다.)

  하지만 사용자님의 말씀처럼 "혼란"이 느껴진다면, 아래와 같은 리팩토링이 필요할 수도 있습니다:

   1. 공통의 Game 인터페이스나 추상 클래스를 만든다.
   2. CuiGame, GuiGame은 오직 입출력 처리(UI)에만 집중하게 하고, 핵심 로직은 별도의 상위 컨트롤러로 분리한다.

  ---

  요약하자면:
  현재 구조는 CuiGame/GuiGame이 메인 컨트롤러(상위) 역할과 UI 상세(하위) 역할을 동시에 수행하려다 보니 경계가 모호해진 상태입니다. CommandParser에서 InputOutput을 제거하는 것은 이러한
  "역할 분리"의 첫걸음이 될 수 있습니다. (파서는 오직 파싱만!)

  이 관점에서 CommandParser의 불필요한 의존성을 먼저 정리해 볼까요? 아니면 전체적인 모듈 구조(DIP)를 더 명확하게 개선하는 방향을 논의해 볼까요?
```

**Q. 거대한 인터페이스가 여러 클라이언트에게 다른 부분만 사용될 때 발생하는 주요 문제는 무엇일까요?**

A. 인터페이스 오염
특정 클라이언트가 사용하지 않는 기능까지 인터페이스에 포함되어 혼란을 주고, 인터페이스 변경 시 무관한 클래스까지 영향을 받는 인터페이스 오염 문제가 발생해요. 이는 인터페이스 분리 원칙(ISP) 위반의 대표적 결과죠.


**Q. 여러 메소드가 특정 인스턴스 변수를 공통으로 접근할 때, 어떤 원칙에 따라 해당 메소드와 변수를 새로운 클래스로 이동하여 리팩토링해야 할까요?**

A. 단일 책임 원칙
하나의 클래스가 여러 책임(변경의 이유)을 가지면 유지보수가 어려워져요. 관련 기능과 데이터를 한 클래스로 모아 책임의 단일성을 높이는 것이 단일 책임 원칙(SRP)의 목표랍니다.


**Q. 핵심 로직이 콘솔과 같은 구체적인 클래스 대신 입출력 인터페이스에 의존하도록 설계하면 어떤 장점이 있을까요?**

A. 테스트 용이성 증가 
구체적인 구현체가 아닌 추상화(인터페이스)에 의존하면, 가짜 객체로 쉽게 교체하여 핵심 로직만 독립적으로 테스트할 수 있어요. 이는 의존성 역전 원칙(DIP) 적용의 주요 이점이죠.


**Q. 클래스가 인터페이스의 일부 메소드에 대해 비어있는 구현(empty method)을 제공하는 경우, 주로 어떤 설계 원칙을 위반할 가능성이 높을까요?**

A. LSP or ISP
클라이언트는 인터페이스 구현체가 인터페이스의 모든 기능을 제공할 것으로 기대하죠. 비어있는 메소드는 기대 위반으로 리스코프 치환 원칙(LSP)이나 인터페이스 분리 원칙(ISP) 위반 가능성을 시사해요.


**Q. CLI와 GUI 환경에서 공통으로 사용되는 게임 실행 로직이 있을 때, 코드 중복을 피하고 새로운 환경에 쉽게 적응하기 위해 어떤 설계 전략을 사용했나요?**

A. 인터페이스를 통한 의존성 분리 및 컴포지션 활용
공통 로직을 별도 클래스로 분리하고, 환경별 차이(게임 루프 방식)를 인터페이스로 추상화한 뒤, 공통 클래스가 이 인터페이스에 의존하도록 컴포지션을 사용했어요. 이는 DRY, DIP, OCP를 만족시키죠.
