
### 9-1. 더 많은 요구사항 추가하기 

서로 다른 구현 클래스가 Carrier 인터페이스의 메서드 일부에만 의존하고 있으면 확장이 어려워진다.
> ISP하고 관련된건가 보다

+플레이어가 보유한 아이템을 임의 위치에 던지는 기능 추가 요청
- 아이템을 던지기 위해서는 
	- 플레이어, 지도, 방이 협력해야 한다 
	- WorldMap도 Carrier를 구현한다 
		- (설계 결함💩) WorldMap 클래스는 Carrier 인터페이스의 일부만 구현한다
		- 구현되지 않는 메서드가 있기 때문에 이를 사용하는 다른 객체와 협력할 수 없다

> 클라이언트 요구사항이 변경될때 인터페이스도 변경이 발생하게 된다. 
> - 관련없는 구현체에도 영향을 끼치겠네 
- 🔎 **여러 클라이언트가 인터페이스의 일부에만 의존하기 때문에 문제 발생**
- ISP 적용 해본다

### 9-2. 인터페이스 분리 원칙
> 날짜 관련 예제 Schedule 사용

**인터페이스 오염(Interface Pollution)**
- 인터페이스에 사용되지 않는 불필요한 오퍼레이션 포함
- 이는 곧 LSP 위반으로 이어진다
	- 변경에 취약한 설계가 들어난다 (= 설계가 취약하다)

> 인터페이스 수정의 원인은 클라이언트라는 사실이 핵심
- `RecurringPlan` 인터페이스를 DateInterval과 Schedule 클래스에서 의존하고 있다
	- 두 클래스에서 인터페이스가 변경된다 
	- 변경될 이유가 여러 개이다

> 해결 방법은 클라이언트 별 인터페이스 분리

>[!info] 인터페이스 분리 원칙 (Interface segregation Principle, ISP)
>**클라이언트가 자신이 사용하지 않는 메소드에 의존하도록 강제되어서는 안 된다.**
>
>어떤 클라이언트가 자신은 사용하지 않지만 다른 클래스가 사용하는 메서드를 포함하는 클래스에 의존할때, 그 클라이언트는 다른 클라이언트가 그 클래스에 가하는 변경에 영향을 받게 된다. 가능하면 이런 결합을 막고 싶다. 따라서 인터페이스를 분리하기를 원한다 - 로버트 C 마틴, 애자일 소프트웨어 개발


> 인터페이스를 여러 개를 하위 모듈에서 구현할 수 있구나 

**의존성 역전 원칙 만족**
- 상위 수준 모듈과 하위 수준 모듈 모두 추상화에 의존

**LSP 만족**
- 클라이언트 관점에서 모든 클래스는 인터페이스를 치환 가능

**개방-폐쇄 원칙 만족**
- 기존 코드 수정 없이 새로운 클래스 추가만으로 기능 확장 가능

**인터페이스는 클라이언트 소유**
> 클라이언트와 인터페이스는 응집도 측면에서 같은 패키지에 있어야 한다

**패키지 수준의 단일 책임 원칙**
- 함께 변경되는 클라이언트와 인터페이스가 같은 패키지에 포함되야 한다✅

**패키지 사이의 의존성 역전 원칙**


**📌 인터페이스 분리 원칙 요약**
- 각 클라이언트가 인터페이스/클래스의 일부 메서드에만 의존한다면
- 변경 시에 영향을 통제할 수 있도록 클라이언트에 따라 인터페이스를 분리하라 
	- 구현 클래스는 하나이더라도, 인터페이스 분리를 통해 노출을 지역화 할 수 있구나 
	- 이를 통해 클래스가 불필요한 이유로 변경되는 것을 줄일 수 있다.



### 9-3. 인터페이스 분리하기 
> 다시 Game 예제로 돌아와 ISP 원칙 적용

**인터페이스 분리 원칙은 인터페이스 관점에서의 단일 책임 원칙**
> 인터페이스를 변경하도록 만드는 클라이언트를 기준으로 인터페이스를 분리하라
- 변경의 이유를 하나만 가지는 인터페이스를 설계할 수 있습니다.


### 9-4. 책임 정리하기 

**기능이 추가될때마다 책임을 재분배할 것 (정리해봐야 할 시점)**

**Game 클래스의 단일 책임 원칙 위반**
- 같은 시점에 같은 이유로 변경되는 로직을 같은 클래스로 모으기
- 도움말 출력 부분이 명령어 추가와 함께 같이 변경된다 



### 9-5. 실행 환경 확장하기 (🔎 되돌아보기 중요)

**되돌아보기** 
처음 Game 클래스는 응집도는 낮고, 결합도는 낮은 상태였다
- 긴 메서드를 동일한 추상화 수준을 가진 작은 메서드들로 분할
- 중복 제거와 명시적인 개념 표현을 위해 값 객체 분리
	- V.O
- 동일한 이유로 변경되는 메서드와 인스턴스 변수를 독립적인 클래스로 분리 
	- SRP
	- 응집도가 높고, 결합도는 낮은 
- 캡슐화 경계를 이용해 변경에 의한 부수효과 통제
	- 캡슐화
- 의존성을 역전시켜 상위 모듈을 변경에 안정적으로 유지
	- SRP, 인터페이스 추출, DIP
- 안정적인 클래스와 분리된 외부 의존성을 가진 불안정한 클래스
	- 외부 의존성은 기술이 변경될 때 교체될 부분

**새로운 요구사항**
- 자바 스윙으로 입출력 처리 추가 

**요구사항이 변경될 때마다 설계 원칙에 따라 리팩터링**
> 목표는 재사용성과 이식성이 높은 설계

**설계 목표**
- CLI와 GUI 환경 모두에서 실행할 수 있도록 개선 

> 🔎 외부 의존성을 고립한 결과 새로운 기능 추가하는게 얼마나 쉬워졌는지 주목한다 


**의존성을 역전시켜 상위 모듈을 변경에 안정적으로 유지**
- 하위 모듈 또한 인터페이스 추상화에 의존하기 때문에 
	- Game을 GuiGame으로 만들어 사용해도 하위 모듈에 영향이 안가는구나



### 9-6. 중복 코드 제거하기

**Q. 거대한 인터페이스가 여러 클라이언트에게 다른 부분만 사용될 때 발생하는 주요 문제는 무엇일까요?**

A. 인터페이스 오염
특정 클라이언트가 사용하지 않는 기능까지 인터페이스에 포함되어 혼란을 주고, 인터페이스 변경 시 무관한 클래스까지 영향을 받는 인터페이스 오염 문제가 발생해요. 이는 인터페이스 분리 원칙(ISP) 위반의 대표적 결과죠.


**Q. 여러 메소드가 특정 인스턴스 변수를 공통으로 접근할 때, 어떤 원칙에 따라 해당 메소드와 변수를 새로운 클래스로 이동하여 리팩토링해야 할까요?**

A. 단일 책임 원칙
하나의 클래스가 여러 책임(변경의 이유)을 가지면 유지보수가 어려워져요. 관련 기능과 데이터를 한 클래스로 모아 책임의 단일성을 높이는 것이 단일 책임 원칙(SRP)의 목표랍니다.


**Q. 핵심 로직이 콘솔과 같은 구체적인 클래스 대신 입출력 인터페이스에 의존하도록 설계하면 어떤 장점이 있을까요?**

A. 테스트 용이성 증가 
구체적인 구현체가 아닌 추상화(인터페이스)에 의존하면, 가짜 객체로 쉽게 교체하여 핵심 로직만 독립적으로 테스트할 수 있어요. 이는 의존성 역전 원칙(DIP) 적용의 주요 이점이죠.


**Q. 클래스가 인터페이스의 일부 메소드에 대해 비어있는 구현(empty method)을 제공하는 경우, 주로 어떤 설계 원칙을 위반할 가능성이 높을까요?**

A. LSP or ISP
클라이언트는 인터페이스 구현체가 인터페이스의 모든 기능을 제공할 것으로 기대하죠. 비어있는 메소드는 기대 위반으로 리스코프 치환 원칙(LSP)이나 인터페이스 분리 원칙(ISP) 위반 가능성을 시사해요.


**Q. CLI와 GUI 환경에서 공통으로 사용되는 게임 실행 로직이 있을 때, 코드 중복을 피하고 새로운 환경에 쉽게 적응하기 위해 어떤 설계 전략을 사용했나요?**

A. 인터페이스를 통한 의존성 분리 및 컴포지션 활용
공통 로직을 별도 클래스로 분리하고, 환경별 차이(게임 루프 방식)를 인터페이스로 추상화한 뒤, 공통 클래스가 이 인터페이스에 의존하도록 컴포지션을 사용했어요. 이는 DRY, DIP, OCP를 만족시키죠.
