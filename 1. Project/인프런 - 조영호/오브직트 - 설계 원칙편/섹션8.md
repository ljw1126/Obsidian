> 코드레벨에서 OCP 설명 👍 (다시보기)

객체 지향에서 코드를 재사용하기 가장 좋은 방법은 상속보다 `합성`이다

### 8-3. 다형적인 개방-폐쇄 원칙  

>[!info] Open-Closed Principle, OCP 개방-폐쇄 원칙
>소프트웨어 개체(클래스, 모듈, 함수 등등)는 **확장에 대해 열려 있어야 하고, 수정에 대해서는 닫혀 있어야 한다.** OCP가 잘 적용된다면, 이미 제대로 동작하고 있던 **원래 코드를 변경하는 것이 아니라 새로운 코드를 덧붙임으로써 변경**할 수 있게 된다.
- 개방 == 확장에 대해 열려 있다. (Open for Extension)
	- 이것은 모듈의 동작을 확장할 수 있다는 것을 의미한다. 애플리케이션의 요구사항이 변경될 때, 이 변경에 맞게 새로운 동작을 추가해 모듈을 확장할 수 있다. 즉, 모듈이 하는 일을 변경할 수 있다. 
- 폐쇄 == 수정에 대해 닫혀 있다. (Closed for Modification)
	- 어떤 모듈의 동작을 확장하는 것이 그 모듈의 소스 코드나 바이너리 코드의 변경으로 이어지는 것은 아니다. 그 모듈의 실행 가능한 바이너리 형태는 링크 가능한 라이브러리, DLL이나 자바의 jar에서도 건드리지 않은 채로 남는다. 

> 핵심은 **안정적인 추상화**를 사용하는 것이다. (통화 시스템 예제 기반 설명)

전통적인 OCP와 다형적인 OCP 비교
- 지난 강의가 전통적인 OCP인데 몇가지 문제를 언급 
	- 업캐스팅 통해 오류 발생 가능에 대해 설명 (이해 덜됨)
	- 📌 전통적인 OCP 목적은 Code Reuse(코드 재사용)이다 
		- 코드를 재사용하기 위한 상속을 `구현 상속(Implementation Inheritance)`이라고 한다.
		- 서브 클래싱
	- 📌 다형적인 OCP 목적은 클라이언트가 다양한 타입의 객체와 협력 지향 (다형성, Polymorphism)
		- 다형성을 위한 `인터페이스 상속(Interface Inheritance)`이라고 한다. 
		- 서브 타이핑
- 전통적인 OCP : 코드 재사용 
	- `Has A` 관계
- 다형적인 OCP : 기능의 확장 의미 
	- `Is A` 관계

> 추상화를 상속받으면 모든 문제가 해결될까?

테스트 코드를 통해 다형적인 OCP의 문제점을 언급 
- 다형적인 개방-폐쇄 원칙은 확장 가능한 코드를 위한 컴파일 타임 원칙
- 그리고 **컴파일 타입에 코드를 확장하면서, 런타임에 안정적으로 실행 가능하도록 하는 원칙이 필요**하다
- 그게 `LSP`이다

### 8-4. 리스코프 치환 원칙 (인터페이스 설명인데 다시 듣기✅)
> LSP 테스트 코드 있음

> 컴파일과 런타임 설계 관점 (다시 듣기)

**IS-A 관계** 
- 클라이언트 관점에서 자식 클래스가 부모 클래스와 호환 가능 
	- `CallCollector is-a Cllector`

**서브타이핑(Subtyping)**
- 클라이언트 관점에서 호환 가능한 두 타입 사이의 관계
- `인터페이스 상속(Interface Inheritance)` 이라고 불림

> LSP 언급, 개념 설명 (다시보기)

>[!info] Liskov Substitution Principle, LSP 리스코프 치환 원칙
>`클라이언트 관점`에서 서브타입(subtype)은 기반 타입(base type)을 `치환`할 수 있어야 한다. - 바바라 리스코프, 데이터 추상화 계층
- 📌자식 클래스가 부모 클래스로 업 캐스팅 되었을때 클라이언트가 부모, 자식 모두에 대해 동일하게 동작하는지가 중요 
	- 다른 방식으로 동작한다면 LSP 위반💩

**올바른 상속 계층에서 모든 클래스들은 리스코프 치환 원칙을 준수**
- 서브 클래스들이 모두 클라이언트 관점에서 기대한대로 동작 

**리스코프 치환 원칙을 위반하는 전형적인 케이스**
- RTTI (Run-Time Type Informaction) -- 💩 code smell 이겠네
	- 조건식에 `instanceof` 로 타입 검사하는 경우
- LSP와 OCP 위반 케이스에 대한 예제 코드 설명 

> ✅ 리스코프 치환 원칙 위반은 잠재적인 개방-폐쇄 원칙을 위반하게 된다
- ✅ 클라이언트 기준으로 치환가능한지 테스트 가능해야 한다.
	- 클라이언트가 정상 동작하는지 확인

**모든 서브 타입들이 클라이언트가 기대하는 행동을 만족하는지 테스트를 만든다** 
- 파라미터 테스트 예제를 통해 LSP를 만족하는지 확인 가능하구나


### 8-5. 리스코프 치환 원칙을 위한 가이드  (상속 문제에 대한 좋은 예제)
> 통화 시스템 예제 확장 // 기본이래 ㅋㅋ

**가이드1. 부모 클래스의 구체 메서드를  오버라이딩 하지 마라**
- super 콜을 피하라

**가이드2. 빈구현으로 오버라이딩 하지 마라**

**가이드3. 예외를 던지는 메서드로 오버라이딩 하지 마라**

**정규화된 계층을 유지하기 위해 리팩터링하라**
- 정규화된 계층(Normalized Hierarchy) : 계층안의 어떤 클래스도 오직 하나의 메서드 구현만 포함 
- 정규화된 계층 만족하면 LSP 만족할 가능성이 높다

> 코드 재사용을 위해 상속을 이용하는 방법을 `차이에 의한 프로그래밍 (Programming by Difference)`
- 💣 JsonRedisReader, CsvRedisReader 생성을 코드 재사용을 위해 상속 사용
	- 정규화되지 않는 계층이다
		- 중복 코드 문제와 조합의 폭발 문제가 발생한다.

상속에서 정규화되지 않은 코드를 사용할 경우 발생하는 두 가지 문제점 
- 점점 더 늘어나는 중복 코드 
- 조합의 폭발적인 증가 (Combinational Explosion) 문제 
	- `기능을 추가할 때마다 클래스의 개수가 폭발적으로 증가하는 문제`
	- **저장소**의 종류와 **데이터 포맷**처럼 서로 다른 변경 이유되는 로직들을 하나의 클래스에 조합해서 구현하기 때문에 발생함 

> 코드를 재사용하기 위해 상속은 좋은 방법이 아니다.
> 
> "Favor Object Composition Over Class Inheritance"
> 클래스 상속보다 객체 합성을 선호하라

> 클래스 다이어그램으로 설명
- AbstractReader
	- read() : 저장소 (파일, 레디스) 종류 별로 달리지는 읽기 로직 
	- parse() : 포맷 (XML, JSON) 종류별로 달라지는 파싱 로직
	- 변경 이유가 두 가지다 💩 분리한다.
- Reader 인터페이스✨
	- 추상 클래스 AbstractReader (Parser와 합성 관계)
	- Parser 인터페이스 

> 단일책임원칙 준수한다 

**확장 가능한 상속 계층**
- 변경의 관점에서 **상속**과 **합성**을 활용
	- 개방-폐쇄 원칙과 
	- 리스코프 치환 원칙을 
	- 만족하는 계층 구조


### 8-6. 아이템 이동 로직 개선하기 (Game 예제)
> 지금까지의 설계 다시 살펴보기
- Game, Player, Room, Transfer
	- Player <- Room 
		- 근데 Room에는 WorldMap이 필요없어서 null 초기화
		- Room이 업캐스팅 되면 npe 발생 (WorldMap이 없어서)
	- Game에 Player 대신 Room 주입하면 대체 불가능 확인 
	- 💩 따라서 Room은 Player를 대체 할 수 없다


Room이 Player를 상속받는다 
하지만 `Room is-not-a Player`

>[!info] LSP (Liskov Substitution Principle)
>(클라이언트 관점에서) 서브 타입(subtype)은 기반 타입 (base type)을 **치환**할 수 있어야 한다

📌 OCP 원칙 위반

> Player의 코드를 재사용하기 위해 상속을 사용했지만,  상속으로 인해 OCP, LSP 원칙을 위반하게 됨
- 현재의 설계를 변경에 취약하다

> 안정적인 추상화에 의존한다
> 의존성 역전 원칙
- Game이 의존해야하는 **추상화**는 무엇인가 ?📌 
	- 리스코프 치환 원칙의 핵심에서 생각해보기 
		- ✅ **클라이언트 관점에서 기대한 행동 수행/정의 해야 한다**
- Game이 기대하는 행동을 `Carrier` 인터페이스로 추상화 


📌**하지만 단일 책임 원칙을 위반하는 Game 클래스**
- 💩 `transfer()`
	- 아이템 이동 로직을 수정할 때 변경
	- 독립적인 테스트가 필요한 private 메서드
- `Transfer` 클래스를 생성 한다 

> 명령-쿼리 분리 원칙을 위반하는 Transfer 클래스 
> - 상태를 변경하는 동시에 값을 반환한다

 >[!info] 명령-쿼리 분리 원칙 (Command-Query Separation Principle, CQS) 
 >객체를 변경하지만 직접적으로 값을 반환하지 않는 `명령(Command)`과 객체에 대한 정보를 반환하지만 변경 하지는 않는 `쿼리(Query)` 간의 명확히 구분해야 한다

> ✅ 명령-쿼리라는게 클래스내에 메서드를 분리하는 것도 가능하구나

> **AbstractCarrier** 추상 클래스를 뽑아서 중복 제거 
- 하지만 추상 메서드가 없다 
- 이는 곧 코드의 재사용을 위한 추상화라는 의미 

>[!info] 상속 원칙
>코드 재사용을 위해 상속을 사용하지 마라✅

> 포워딩 클래스(Forwarding Class)와 **포워딩** 메서드(Forwarding Method)
- 상속안에서만 중복코드가 재사용된다면 상속을 사용하기도 하신단다 (ex. 포워딩 클래스)

### 같은 패키지에 하위 뎁스에 계층을 나누는거랑, 상위 패키지 뎁스 기준으로 나누는  방식의 차이에 대해

```shell
✦ 패키지 구조를 설계할 때 상위 패키지가 하위 패키지에 의존하지 않도록 인터페이스를 상위로 올리는 것은 의존성 역전 원칙(DIP)을 패키지 수준에서 아주 잘 적용하신 사례입니다.

  말씀하신 "하위 뎁스로 나눌지, 같은 뎁스로 나눌지"에 대한 고민은 실무에서도 매우 빈번하며, 보통 '응집도'와 '가시성(Visibility)', 그리고 '도메인 지식의 전파' 관점에서 기준을 잡습니다.

  주요 기준들을 정리해 드립니다.

  1. 계층형 vs 기능형 (Package by Layer vs Package by Feature)
   * 하위 뎁스로 구성 (Package by Layer 지향):
       * adventure.game, adventure.console 처럼 나누는 방식입니다.
       * 기준: adventure라는 하나의 큰 컨텍스트 안에서 기술적인 역할(UI, Core Logic, Infrastructure)을 분리하고 싶을 때 사용합니다.
       * 장점: adventure 전체를 하나의 모듈처럼 다루기 쉽고, 패키지 간의 위계가 명확해집니다.
   * 같은 뎁스로 구성 (Package by Feature 지향):
       * org.eternity.adventure, org.eternity.console 처럼 동등한 위치에 두는 방식입니다.
       * 기준: 각각이 독립적인 서비스나 라이브러리로 분리될 가능성이 높을 때 사용합니다.
       * 장점: 모듈 간 결합도가 낮아지고, 나중에 별도의 프로젝트나 모듈로 떼어내기가 훨씬 수월합니다.

  2. 의존성 역전과 가시성 (Visibility)
  현재 구조처럼 InputOutput 인터페이스는 adventure에 두고, 구현체인 Console은 adventure.console에 두는 방식은 매우 견고한 설계입니다.

   * 하위 패키지로 둘 때의 이점: Java의 protected나 package-private(default) 접근 제한자를 활용하여, 외부(다른 상위 패키지)에는 구현체를 감추고 인터페이스만 노출하기에 유리합니다.
   * 같은 뎁스로 둘 때의 위험: 동등한 위치에 있으면 모든 것을 public으로 열어야 하는 경우가 많아져, 의도치 않은 의존성이 생길 확률이 높아집니다.

  3. 추천하는 선택 기준 (The Common Reuse Principle)
  로버트 C. 마틴(Uncle Bob)의 공통 재사용 원칙(CRP)을 참고하면 좋습니다.

   4. 함께 배포되어야 하는가? adventure가 없으면 console이 아무 의미가 없다면 하위 패키지로 둡니다.
   5. 도메인이 중심인가? adventure가 이 프로젝트의 핵심 도메인이고 나머지가 이를 지원하는 형태라면, 도메인을 최상위(또는 중심)에 두고 나머지를 하위나 별도 패키지로 격리합니다.

  현재 구조에 대한 의견
  현재 org.eternity.adventure 하위에 game, console 등을 두신 것은 "Adventure 게임이라는 하나의 문맥(Bounded Context)"을 명확히 보여주는 아주 좋은 구조입니다.

   * 인터페이스(Policy): org.eternity.adventure (고수준 정책)
   * 구현체(Detail): org.eternity.adventure.console, org.eternity.adventure.game.command (저수준 상세)

  이렇게 구성하면 고수준 정책이 저수준 상세의 변경에 영향을 받지 않는 깔끔한 구조가 유지됩니다. 만약 console 외에 web UI가 추가된다면 org.eternity.adventure.web 패키지를 추가하기만 하면
  되므로 확장성 면에서도 뛰어납니다.

  결론적으로: 지금처럼 논리적인 포함 관계가 성립한다면 하위 패키지로 계층화하는 것이 "이 코드가 무엇을 위한 코드인가"를 설명하는 데 더 효과적입니다. 같은 뎁스로 나누는 것은 두 패키지가
  서로 다른 생명주기를 가질 때 고려해 보세요.
```

**Q. 전통적인 개방-폐쇄 원칙(OCP)과 다형적인 OCP의 주요 차이점은 무엇일까요?**

A. 다형적인 OCP는 추상화(인터페이스 등)를 통해 기능을 확장하며 기존 코드는 수정하지 않습니다. 반면 전통적인 OCP는 상속을 활용해 부모 클래스의 코드를 재사용하며 차이점만 구현하는 데 초점을 맞춥니다.


**Q. 리스코프 치환 원칙(LSP)이 객체 지향 설계에서 주로 강조하는 것은 무엇일까요?**

A . LSP는 부모 타입의 객체를 자식 타입의 객체로 치환했을 때, 부모 타입을 사용하는 클라이언트의 행동이 변하지 않아야 한다는 원칙입니다. 이는 클라이언트 관점에서의 행위적 호환성을 의미하죠.


**Q. 강의에서 상속을 코드 재사용의 주된 목적으로만 사용하는 것이 문제될 수 있다고 설명합니다. 그 이유는 무엇일까요?**

A. 상속을 코드 재사용만을 위해 사용하면 자식 클래스가 부모 타입으로 업캐스팅되어 예상치 못한 클라이언트에게 사용될 수 있습니다. 이때 자식 클래스의 다른 동작이 클라이언트의 기대를 깨뜨려 런타임 문제를 일으킬 수 있습니다.


**Q. 텍스트 어드벤처 게임 예제에서 Player와 Room 클래스의 아이템 이동 로직을 개선하며 Carrier 인터페이스를 도입한 주된 목적은 무엇일까요?**

A. Carrier 인터페이스는 아이템을 담고(add), 찾고(find), 제거(remove)하는 객체들의 공통 행동을 추상화했습니다. 이를 통해 게임 클래스는 구체적인 Player나 Room 대신 Carrier 추상화에 의존하게 되어 DIP를 만족시키고 LSP 위반 문제를 해결했죠


**Q. 강의에서 코드 재사용을 위해 상속보다 객체 합성을 더 권장하는 대표적인 상황은 언제인가요?**

A. 데이터 저장 방식(파일, Redis)과 데이터 형식(CSV, JSON)처럼 여러 독립적인 이유로 코드가 변경될 때, 상속은 조합 폭발을 일으키고 코드를 중복시킬 수 있습니다. 이때 각 변경의 축을 별도의 클래스로 분리하고 합성을 통해 조합하는 것이 더 유연하고 재사용성이 높습니다.

  1. Carrier 인터페이스의 default 메서드
  default 메서드는 인터페이스에서 공통으로 사용될 로직을 미리 구현해 두는 용도가 맞습니다. 이를 통해 얻는 이점은 다음과 같습니다.

   * 코드 중복 제거: Inventory, Room 등 Carrier를 구현하는 여러 클래스에서 반복될 "아이템 포함 여부 확인", "아이템 찾기" 같은 공통 로직을 한 곳에 모을 수 있습니다.
   * 유연한 확장: 기존 인터페이스에 새로운 기능을 추가해도, 이를 구현하는 클래스들을 모두 수정할 필요가 없어 하위 호환성을 유지하기 좋습니다.
   * 선택적 오버라이딩: 말씀하신 대로 기본 구현을 그대로 써도 되고, 특정 클래스에서 성능 최적화 등이 필요하다면 해당 메서드만 따로 재정의(Override)할 수 있습니다.

  실제로 Carrier.java를 보면 items()를 통해 아이템 목록을 가져와 처리하는 공통 로직들이 default로 잘 정의되어 있을 것입니다.

  2. Transfer 클래스와 합성(Composition)
  Transfer 클래스는 합성(Composition)을 아주 잘 활용한 사례입니다.

   * 책임: Transfer는 이름 그대로 한 Carrier(출발지)에서 다른 Carrier(목적지)로 아이템을 안전하게 옮기는 책임을 가집니다.
   * 합성 기반: Transfer는 특정 Carrier를 상속받는 것이 아니라, 생성자 등을 통해 두 개의 Carrier 객체를 주입받아 사용합니다.
       * source.remove(item) -> target.add(item) 처럼 주입받은 객체들에게 메시지를 보내 협력하는 구조입니다.
   * 장점: 이렇게 합성을 사용하면 Inventory에서 Room으로, Room에서 Inventory로, 혹은 Room 간의 이동 등 어떤 Carrier 구현체 사이에서도 아이템을 옮길 수 있는 범용성을 갖게 됩니다.

  즉, "무엇인가(Carrier)로부터 무엇인가(Carrier)로 아이템을 옮긴다"는 행위 자체를 독립된 객체로 분리해낸 훌륭한 설계라고 볼 수 있습니다.

  혹시 구체적인 코드 로직 중에서 더 궁금한 부분이 있으신가요? Transfer의 구현 방식이나 Carrier의 다른 메서드들에 대해서도 더 자세히 살펴봐 드릴 수 있습니다.
