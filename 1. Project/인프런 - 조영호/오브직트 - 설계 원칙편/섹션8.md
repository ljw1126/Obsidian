
### 8-1. 새로운 요구사항 추가하기 
- 아이템 줍기 (`take`)
- 아이템 버리기 (`drop`)
- 현재 보유 아이템 목록 출력하기 (`inventory`)

> 아이템을 줍거나, 버리기를 할때 `방`과 `풀레이어`의 행위가 중복된다 (= 추상화 가능)

**코드를 재사용하는 전통적인 객체 지향 기법**
> 상속 (Inheritance)

Player 를 상속 <- Room 

**업캐스팅(Upcasting)**
- 자시 클래스가 부모 클래스 치환 가능 
	- 1. 부모 클래스 타입 변수에 자식 클래스 인스턴스 대입 
	- 2. 부모 클래스 타입 파라미터에 자식 클래스 인스턴스 전달 
	- 3. 리턴 타입이 부모 클래스 타입인 경우 자식 클래스 인스턴스 반환


**하지만 상속으로 인해 발생하는 문제점**
- Room이 필요 없는 곳에서도 Room이 Player로 업캐스팅해서 주입 가능하다. 
	- 그러다보니 Room에 없는 move() 메서드 호출시 npe 발생 
		- 1. `instanceof`로 타입을 검사하는 방법으로 보호 구문 작성 가능 
			- 그렇지만 부모 클래스가 자식 클래스의 타입을 알 필요가 있을까?
		- 2. 자식 클래스에서 오버라이딩 메서드를 만든다. 
			- 그렇지만 유지보수 관점에서 사용하지 않는 메서드를 구현하는건 좋지 않다

### 8-2. 전통적인 개방-폐쇄 원칙 (OCP) 

> 통화 시스템에 요구사항 추가에 대한 케이스 설명 

상속에서 부모 클래스와 다른 부분만 자식 클래스 안에 구현해서 기능 확장하는 방법을 차이에 의한 프로그래밍(`Programming By difference`)

> 코드레벨에서 OCP 설명 👍 (다시보기)
- 기존 클라이언트에 영향을 주지 않고 확장 가능

> 상속의 업캐스팅으로 인한 이슈 설명 

>[!info] 상속 원칙
>코드 재사용을 위해 상속을 사용하지 마라

객체 지향에서 코드를 재사용하기 가장 좋은 방법은 상속보다 합성이다

> Favor Object Composition Over Class Inheritance 
> 클래스 상속보다 객체 합성을 선호하라

> 다형적인 개방-폐쇄 원칙의 등장

### 8-3. 다형적인 개방-폐쇄 원칙 

전통적인 OCP와 다형적인 OCP 비교
- 지난 강의가 전통적인 OCP인데 몇가지 문제를 언급 
- 전통적인 OCP : 코드 재사용 
	- `Has A` 관계
- 다형적인 OCP : 기능의 확장 의미 
	- `Is A` 관계

테스트 코드를 통해 다형적인 OCP의 문제점을 언급 
- 그리고 컴파일 타입에 코드를 확장하면서, 런타임에 안정적으로 실행 가능하도록 하는 원칙이 필요하다
- 그게 `LSP`이다
### 8-4. 리스코프 치환 원칙 

> 컴파일과 런타임 설계 관점 (다시 듣기)

**IS-A 관계** 
- 클라이언트 관점에서 자식 클래스가 부모 클래스와 호환 가능 
	- `CallCollector is-a Cllector`

**서브타이핑(Subtyping)**
- 클라이언트 관점에서 호환 가능한 두 타입 사이의 관계
- `인터페이스 상속(Interface Inheritance)` 이라고 불림
