
### 8-1. 새로운 요구사항 추가하기 
- 아이템 줍기 (`take`)
- 아이템 버리기 (`drop`)
- 현재 보유 아이템 목록 출력하기 (`inventory`)

> 아이템을 줍거나, 버리기를 할때 `방`과 `풀레이어`의 행위가 중복된다 (= 추상화 가능)

**코드를 재사용하는 전통적인 객체 지향 기법**
> 상속 (Inheritance)

Player 를 상속 <- Room 

**업캐스팅(Upcasting)**
- 자시 클래스가 부모 클래스 치환 가능 
	- 1. 부모 클래스 타입 변수에 자식 클래스 인스턴스 대입 
	- 2. 부모 클래스 타입 파라미터에 자식 클래스 인스턴스 전달 
	- 3. 리턴 타입이 부모 클래스 타입인 경우 자식 클래스 인스턴스 반환


**하지만 상속으로 인해 발생하는 문제점**
- Room이 필요 없는 곳에서도 Room이 Player로 업캐스팅해서 주입 가능하다. 
	- 그러다보니 Room에 없는 move() 메서드 호출시 npe 발생 
		- 1. `instanceof`로 타입을 검사하는 방법으로 보호 구문 작성 가능 
			- 그렇지만 부모 클래스가 자식 클래스의 타입을 알 필요가 있을까?
		- 2. 자식 클래스에서 오버라이딩 메서드를 만든다. 
			- 그렇지만 유지보수 관점에서 사용하지 않는 메서드를 구현하는건 좋지 않다

### 8-2. 전통적인 개방-폐쇄 원칙 (OCP) 

> 통화 시스템에 요구사항 추가에 대한 케이스 설명 

상속에서 부모 클래스와 다른 부분만 자식 클래스 안에 구현해서 기능 확장하는 방법을 차이에 의한 프로그래밍(`Programming By difference`)

> 코드레벨에서 OCP 설명 👍 (다시보기)
- 기존 클라이언트에 영향을 주지 않고 확장 가능

> 상속의 업캐스팅으로 인한 이슈 설명 

>[!info] 상속 원칙
>코드 재사용을 위해 상속을 사용하지 마라

객체 지향에서 코드를 재사용하기 가장 좋은 방법은 상속보다 합성이다

> Favor Object Composition Over Class Inheritance 
> 클래스 상속보다 객체 합성을 선호하라

> 다형적인 개방-폐쇄 원칙의 등장

### 8-3. 다형적인 개방-폐쇄 원칙 

전통적인 OCP와 다형적인 OCP 비교
- 지난 강의가 전통적인 OCP인데 몇가지 문제를 언급 
- 전통적인 OCP : 코드 재사용 
	- `Has A` 관계
- 다형적인 OCP : 기능의 확장 의미 
	- `Is A` 관계

테스트 코드를 통해 다형적인 OCP의 문제점을 언급 
- 그리고 컴파일 타입에 코드를 확장하면서, 런타임에 안정적으로 실행 가능하도록 하는 원칙이 필요하다
- 그게 `LSP`이다
### 8-4. 리스코프 치환 원칙 

> 컴파일과 런타임 설계 관점 (다시 듣기)

**IS-A 관계** 
- 클라이언트 관점에서 자식 클래스가 부모 클래스와 호환 가능 
	- `CallCollector is-a Cllector`

**서브타이핑(Subtyping)**
- 클라이언트 관점에서 호환 가능한 두 타입 사이의 관계
- `인터페이스 상속(Interface Inheritance)` 이라고 불림

> LSP 언급, 개념 설명 (다시보기)

**올바른 상속 계층에서 모든 클래스들은 리스코프 치환 원칙을 준수**
- 서브 클래스들이 모두 클라이언트 관점에서 기대한대로 동작 

**리스코프 치환 원칙을 위반하는 전형적인 케이스**
- RTTI 
	- `instanceof` 로 타입 검사하는 경우

**모든 서브 타입들이 클라이언트가 기대하는 행동을 만족하는지 테스트를 만든다** 
- 파라미터 테스트 예제를 통해 LSP를 만족하는지 확인 가능하구나


### 8-5. 리스코프 치환 원칙을 위한 가이드 

**부모 클래스의 구체 메서드를 오버라이딩 하지 마라**
- super 콜을 피하라

**빈구현으로 오버라이딩 하지 마라**

**예외를 던지는 메서드로 오버라이딩 하지 마라**

**정규화된 계층을 유지하기 위해 리팩터링하라**

> 코드 재사용을 위해 상속을 이용하는 방법을 `차이에 의한 프로그래밍 (Programming by Difference)`

상속에서 정규화되지 않은 코드를 사용할 경우 발생하는 두 가지 문제점 
- 점점 더 늘어나는 중복 코드 
- 조합의 폭발적인 증가 (Combinational Explosion) 문제 
	- `기능을 추가할 때마다 클래스의 개수가 폭발적으로 증가하는 문제`
	- **저장소**의 종류와 **데이터 포맷**처럼 서로 다른 변경 이유되는 로직들을 하나의 클래스에 조합해서 구현하기 때문에 발생함 

> 코드를 재사용하기 위해 상속은 좋은 방법이 아니다.
> 
> "Favor Object Composition Over Class Inheritance"
> 클래스 상속보다 객체 합성을 선호하라

> 클래스 다이어그램으로 설명

**확장 가능한 상속 계층**
- 변경의 관점에서 상속과 합성을 활용
	- 개방-폐쇄 원치고가 
	- 리스코프 치환 원칙을 
	- 만족하는 계층 구조


### 8-6. 아이템 이동 로직 개선하기
> 지금까지의 설계 다시 살펴보기

Room이 Player를 상속받는다 
하지만 `Room is-not-a Player`

>[!info] LSP (Liskov Substitution Principle)
>(클라이언트 관점에서) 서브 타입(subtype)은 기반 타입 (base type)을 **치환**할 수 있어야 한다

OCP 원칙 위반

> Player의 코드를 재사용하기 위해 상속을 사용했지만,  상속으로 인해 OCP, LSP 원칙을 위반하게 됨
- 현재의 설계를 변경에 취약하다

> 의존성 역전 원칙
- Game이 의존해야하는 추상화는 무엇인가 ?📌
	- 리스코프 치환 원칙의 핵심에서 생각해보기 
		- **클라이언트 관점에서 기대한 행동 수행/정의 해야 한다**
- Game이 기대하는 행동을 Carrier 인터페이스로 추상화 


**하지만 단일 책임 원칙을 위반하는 Game 클래스**
- 💩 `transfer()`
	- 아이템 이동 로직을 수정할 때 변경
	- 독립적인 테스트가 필요한 private 메서드
- Transfer 클래스를 생성 한다 

> 명령-쿼리 분리 원칙을 위반하는 Transfer 클래스 
> - 상태를 변경하는 동시에 값을 반환한다

 >[!info] 명령-쿼리 분리 원칙 (Command-Query Separation Principle, CQS) 
 >객체를 변경하지만 직접적으로 값을 반환하지 않는 `명령(Command)`과 객체에 대한 정보를 반환하지만 변경 하지는 않는 `쿼리(Query)` 간의 명확히 구분해야 한다

> 명령-쿼리라는게 클래스내에 메서드를 분리하는 것도 가능하구나

> AbstractCarrier 추상 클래스를 뽑아서 중복 제거 
- 하지만 추상 메서드가 없다 
- 이는 곧 코드의 재사용을 위한 추상화라는 의미 

>[!info] 상속 원칙
>코드 재사용을 위해 상속을 사용하지 마라✅

> 포워딩 클래스(Forwarding Class)와 포워딩 메서드(Forawrding Method)
- 상속안에서만 중복코드가 재사용된다면 상속을 사용하기도 하신단다 (ex. 포워딩 클래스)
