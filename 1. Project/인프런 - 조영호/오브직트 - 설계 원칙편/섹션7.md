
>[!note] 책임을 한 문장으로 적어본다

>

>[!note] 인스턴스 변수와 메서드 사이의 관계를 살펴본다
>일부 인스턴스 변수가 일부 메서드에 의해서만 사용되는지 살펴본다

>[!note]  서로 배타적으로 초기화되는 인스턴스 변수가 있는지 살펴본다
>어떤 변수들이 초기화될 때 함께 초기화되지 않는 인스턴스 변수들이 있는지 살펴본다

>[!note] 테스트하고 싶은 private 메서드가 있는지 살펴본다
>너무 많은 private (또는 protected) 메서드가 왔을때 테스트하고 싶은 private 메서드가 존재하는지 살펴본다.

>데이터베이스 연결이나 외부 시스템과의 연동 등과 같이 외부에 위치하는 불안정한 의존성을 찾는다.



### 7-1. 외부 의존성과 테스트

- Console이 Scanner, System에 의존함
- 이 영향이 Game까지 전파되어 간접적으로 의존하게 된다
	- 구현체에 직접 의존해서 그런건가 싶음 🤔 
	- Console 내부에 숨겨진 의존성으로 인해 

>[!note] 테스트를 어렵게 만드는 의존성
>구체 클래스에 대한 의존성, 숨겨진 의존성

키보드 입력과 화면 출력을 처리하는 `저수준 메커니즘` 의 변경이
게임의 전체적인 흐름을 조율하는 `고수준 정책`에 영향 전파 가능한 구조이다
- 고수준 정책 <- 저수준 정책 변경의 영향이 💩
- 변경의 영향이 고수준 정책 -> 저수준 메커니즘 ✅
	- **DIP**를 사용한다. 
### 7-2. 의존성 역전 원칙 - 상위 수준과 하위 수준  (첫번째 관점)

>[!info] 의존성 역전 원칙 (Dependency Inversion Principle, DIP) 
>- 상위 수준 모듈은 하위 수준 모듈에 의존해서는 안된다. 둘 모두 추상화에 의존해야 한다.
>- 추상화는 세부 사항에 의존해서는 안된다. 세부 사항이 추상화에 의존해야 한다.
> 
> 로버트 C. 마틴, 애자일 소프트웨어 개발

> 통화 관리 시스템 예제 
> - Call 클래스
> - CsvReader 클래스
> - CallHistory 클래스
> - CallCollector 클래스
> - JsonReader 클래스 추가
> - Reader 인터페이스 추출


**협력 관점에서 분석**
- CSV 파일에서 Call을 읽는 로직 
	- 통화 시간을 계산하는 본질적인 문제와는 거리가 먼 파일 처리 로직

본질적인 목적과 관련 깊은 부분을 `상위 수준 모듈`이라고 한다
- 통화 시간을 계산하는 본질적인 문제를 처리하는 `상위 수준 모듈`

CallCollector가 CsvReader를 직접 생성한다. 
그러다보니 CallCollector (상위 수준 모듈)이 하위 수준 모듈(Reader) 변경 영향을 받는다.
- Reader 인터페이스 추출하고 상위수준 모듈이 의존하도록 한다. 

생성자에서 하위 모듈을 직접 생성하지 않고, 제 3의 객체가 하위 수준 모듈 생성하도록한다. 
- 이를 통해 상위 수준 모듈에서 하위 수준 모듈에 대한 의존성 제거 

이렇게 외부의 객체가(제 3의 객체)가 의존성을 해결해주는 것을 
- `의존성 주입 (Dependency Injection)`
	- 종류 
		- 생성자 주입 방식 (선호하심✨)
		- setter 주입 방식
		- 메서드 주입 방식 
			- 메서드의 인자를 이용해서 의존성 주입 

**두 가지 책임의 분리**
> 객체를 사용하는 책임과 생성하는 책임이 두 가지 분리 된다
> - 의존성 주입할 객체 **생성**은 외부의 제3의 객체가 담당
> - 의존성 주입된 객체를 **사용**하는 책임은 상위 수준 모듈이 담당


**결합도를 낮추기 위해 생성과 사용 분리** ✅
- 객체는 다른 객체를 생성하기만 하거나 사용하기만 해야 한다
- `생성 사용 분리 (Separating Use from Construction)`

>의존성 역전 원칙의 핵심은 `추상화`다
>
>`구현이 아닌 인터페이스에 대해 프로그래밍하라 (Program to an interface, not implementation)`


**요약**
- 하위 수준 모듈때문에 불안정한 상위 수준 모듈이 있을 때
- 하위 수준 모듈을 추상화와 구현으로 분리 
- 상위 수준 모듈이 추상화에 의존하도록 수정 
- 이렇게 의존성을 역전 시키면 
	- 상위 수준 모듈을 수정하지 않고도 하위 수준 모듈 교체 가능 

### 7-3. 의존성 역전 원칙 - 추상화와 세부 사항  (두번째 관점, 패키지 관점에서 설명✅)
> (DIP)
> 추상화는 세부 사항에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다.

> 소프트웨어 설계에서 추상화란 
> - 자주 변하지 않는 안정적인 부분을 말함
> - 불필요한 측면(자주 변하는) 제외하고 필요한 측면(자주 변하지 않는)만 남긴다. 

> 세부사항은 
> - 자주 변하는 불안정한 부분


> [!info] 의존성 역전 원칙을 적용할때는 클래스 뿐만아니라 패키지 수준에서도 의존성을 조정해야 합니다.

**Q. 강의 초반부 게임 클래스 테스트 코드를 이해하고 관리하기 어려웠던 주요 원인은 무엇일까요?**

A. 초기 게임 클래스는 System.in, System.out 등 외부 입출력 의존성을 내부에 숨기고 있어, 테스트 시 외부에서 이를 제어하거나 결과를 확인하기 어려웠어요.


**Q. 상위 수준 모듈이 하위 수준의 구체적인 모듈에 직접 의존할 때 발생하는 주요 문제는 무엇인가요?**

A. 상위 모듈이 하위 구체 클래스에 의존하는 구조에서는 하위 모듈의 작은 변경이 상위 모듈에 전파되어 수정이 어려워지고 설계가 불안정해져요.


**Q. 의존성 역전 원칙(DIP)이 상위/하위 수준 모듈의 의존성에 대해 말하는 핵심은 무엇일까요?**

A. DIP는 고수준 모듈과 저수준 모듈 모두 구체 클래스가 아닌 인터페이스나 추상 클래스 같은 추상화에 의존해야 한다고 말하며 유연성을 높여줘요.


**Q. 의존성 주입(DI)이란 무엇이며, DIP를 적용한 설계에서 중요한 역할은 무엇인가요?**

A. DI는 모듈이 내부에서 의존 객체를 직접 생성하지 않고 외부에서 주입받는 방식으로, DIP로 추상화에 의존하게 된 모듈에 실제 구현체를 연결하며 결합도를 낮춰요


**Q. 의존성 역전 원칙(DIP)과 의존성 주입(DI)을 적용하면 코드 테스트 용이성이 어떻게 향상될까요?**

A . DIP/DI로 의존성이 분리되면, 테스트 시 실제 외부 의존성 대신 테스트용 가짜 객체(Fake Object)를 주입하여 입력/출력을 쉽게 제어하며 결과를 명확히 확인할 수 있어요.
