
### 6-1. 디미터 법칙과 묻지 말고 시켜라 원칙  (Game 클래스 예제)

**디미터 법칙**
> 협력하는 객체의 내부 구조를 모르고도 협력이 가능해야 함
> 결합도를 낮추기 위해 메시지 전송 대상을 제한해야 한다
- 구조 변경을 캡슐화하게 된다 

>[!info] 디미터 법칙 (Law of Demeter, LoD)
> 클래스 내부의 메서드는 아래 조건을 만족하는 인스턴스에만 메시지를 전송해야만 한다. 
>- this 객체
>- 메서드의 매개변수
>- this의 속성
>- this의 속성인 컬렉션의 요소
>- 메서드 내에서 생성된 지역 객체 
>  (크레이그 라만, UML과 패턴의 적용)


디미터 법칙을 따르는 클래스 (코드 설명)

**디미터의 다른 표현**
- 낯선 자에게 말하지 말라
	- don't talk to strangers
- 오직 인접한 이웃에게만 말하라
	- only talk to your immediate neighbors
- 오직 하나의 도트만 사용하라 
	- use only one dot

**디미터 법칙을 준수하는 코드를** 
- `부끄럼타는 코드(shy code)`라고도 부른다

> 디미터 법칙을 위반하는 Game 클래스

```java
// 열차 충돌 (Train Wreck)
player.position().shift(direction);
```


>[!note] 묻지 말고 시켜라 (Tell, don't Ask, TDA)
>**여러분은 객체엑 원하는 작업을 이야기함으로써 명령을 전송해야 합니다.** 우리는 객체의 상태를 묻고, 결정을 내린 후에, 객체에게 무엇을 수행할지 지시하는 방식을 선호하지 않습니다. 
>
>호출자는 호출된 객체의 상태를 바탕으로 결정을 내린 후에 해당 객체의 상태를 변경해서는 안됩니다. **구현하고 있는 논리는 아마도 호출된 객체의 책임이지, 호출하는 쪽의 책임은 아닐 것입니다.** 객체 외부에서 결정을 내리는 것은 객체의 캡슐화를 위반하며, 버그가 발생하기 쉬운 환경을 만들게 됩니다. - 앤드류 헌트 & 데이트 토마스, 실용주의 프로그래머


**해결 방법**
- `책임의 이동 (Shift or responsibility)`
	- Player가 의사결정 하도록한다.
		- tryMove, showRoom 메서드 리팩어링한다

> Player의 내부 구조가 변경되더라도 Game이 변경되지 않도록 캡슐화를 했기때문에 showRoom에서는 디미터 법칙 위반이 아니게 된다. (코드로 잘 보자🔎)

**따라서 가끔은 물어봐도 된다.**
- 언제 ? 내부 구조를 노출시키지 않는다면 가능

### 6-2. 명령 쿼리 분리 원칙으로 부수효과 관리하기 (Schedule 클래스 예제)

**루틴(Routine)**
- 일련의 절차를 묶어 호출 가능하도록 이름을 부여한 기능 모듈
- 부수효과와 반환값의 유무에 따라 프로시저와 함수로 분류
	- **프로시저 (Procedure)**
		- 부수효과 존재, 반환값이 없음 
		- `void name(value)`
		- **Command(명령)** 에 해당
	- **함수 (Function)**
		- 부수효과 없음, 반환 값 존재
		- `String name()`
		- **Query(쿼리)** 에 해당

> [!note] 명령-쿼리 분리 원칙 (Command-Query Separation Prinicple, CQS)
> 부수효과를 발생시키지 않는 것만을 함수로 제한함으로써 소프트웨어에서 말하는 "함수"의 개념이 일반 수학에서의 개념과 상충되지 않게 한다.
> 
> **객체를 변경하지만 직접적으로 값을 반환하지 않는 명령(Command)과 객체에 대한 정보를 변환하지만 변경하지는 않는 쿼리(Query)를 명확히 구분해야 한다.**
> -버트란드 마이어, 객체지향 소프트웨어 구축


**질문이 답변을 수정해서는 안된다.** 
- 📌 객체의 상태를 변경하는 명령은 반환 값을 가질 수 없다.
- 📌 객체의 정보를 반환하는 쿼리는 상태를 변경할 수 없다.


**명령과 쿼리를 분리해야 하는 이유**
- 이해하기 쉽게 하기 위해

> Schedule 클래스 예제로 설명하심
- 단일 책임원칙을 준수 중
- 새로운 요구사항 추가

> [!warning]  명령과 쿼리 두가지 특성을 가지는 함수는 코드 내부를 살펴보기 전까지는 결과 예측이 어렵다


**일관성 있는 메서드 시그니처**
명령과 쿼리를 분리하면 실행 결과가 예측하기 쉽게 만들어진다


**명령과 쿼리의 메서드 시그니처**
- 쿼리 
	- 부수 효과 없음
	- 몇번을 연이어 호출해도 결과 동일해야 함
	- ex. `boolean includes(LocalDate day);`
- 명령
	- 부수효과 발생
	- 호출할때마다 상태 변경
	- ex. `void schedule(LocalDate day);`
- 🔎 **메서드 시그니처만으로 부수효과의 여부 판단 가능**
	- 아.. 반환값이 있으면 쿼리고, 없으면 명령이다..

> 명령 쿼리 분리 원칙과 값객체를 활용하는 방법이 이해 덜됨🔎

> 다시 Player 클래스 예제
- move 메서드에 명령과 쿼리가 혼합되어 있다. 
- 그러다보니 부수효과가 테스트 과정에서 발생한다 
	- 부수효과가 뭔지 알기 위해 메서드 내부 구현 확인이 필요해진다 
- canMove(쿼리)와 move(명령) 메서드 분리해서 부수효과 통제해서 실행 결과를 이해하기 쉽게, 재사용하기 쉽게 만들어준다
