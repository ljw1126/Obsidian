### 4-1. 참조 객체와 별칭 문제

클래스 작게 만드는 두가지 방법
- 값 객체 분리
- 단일 책임 원칙 적용

> 참조 객체 문제를 보완하기 위해 값 객체를 사용한다.

각 게임별 매출 관리

> 테스트 코드 설명한다.

영업 이익을 정상적으로 계산하기 위해서는 
- 하나의 Sales 객체를 사요해야 한다.

**참조 객체 (Reference Object)**
- 모든 클라이언트가 동일한 객체를 이용해서 오퍼레이션을 실행 
- 일반적으로 도메인 상의 중요한 개념을 표현 
- 가변 객체 (Mututal Object로 구현)
  - Sales 


**동일성 (identical)**
- 모든 변수가 동일한 객체 참조 

별칭 (Aliasing)
- 동일한 객체를 가리키는 서로 다른 이름의 객체 참조
   - 참조 객체의 이름을 서로 다르게 지정하여 사용 가능

별칭을 통한 부수효과 
- 하나의 참조를 통해 발생한 변경이 다른 참조로 전파

**별칭은 양날의 검**
- (good) 두 참조가 동일한 상태 공유 가능 
- (bad) 원하지 않는 부수효과 발생 // 버그 🐞

참조 객체의 별칭 문제로 인한 복잡성을 감소시킬 수 있는 방법은?
> 값 객체 (Value Object)를 사용한다. 이를 통해 참조 객체의 복잡성을 낮출수 있다. 


### 4-2. 값 객체의 가치 

**값 객체 (Value Object)**
- 값이 동일하면 동일한 객체로 취급 
- 값 비교를 통해 참조 객체의 복잡성 감소 
- 불변 객체 (immutable object)로 구현

상태가 변하지 않는 Money 값 객체 
- 불변 객체 (Immutable Object)

> 값 객체를 이용해서 참조 객체의 복잡성을 낮출 수 있다. // 사이드 이펙트도, 다만 객체 생성 비용은 증가

> 가변 객체와 불변 객체의 계산 예시 (Money)
- 가변 객체는 순차적으로 코드를 다 이해해야 한다 
  - 상태 변경을 예측하기 어렵고, 별칭까지 사용하면 실행 결과를 보장하기 어렵다
- 불변 객체는 객체 생성 단계의 값만 기억하면 된다. 

**값 객체를 이용한 단순화**
- 상태 추적과 예상하지 못한 부수효과에 대한 걱정 없이 사용


| **참조 객체**           | **값 객체**           |
| ------------------- | ------------------ |
| 식별자를 기반으로 객체 동일성 체크 | 속성을 기반으로 객체 동등성 체크 |
| 가변 객체로 구현           | 불변 객체로 구현          |
| 별칭 문제 발생 (사이드 이펙트)  | 별칭 문제 방지           |

**값 객체의 가치 (사용 이유)**
- 참조 객체 안에 숨겨진 모호한 개념을 명시적으로 드러내서 복잡성 감소
- 참조 객체 안의 복잡한 로직이나 중복 코드를 값 객체로 이동

**✅ 암시적인 개념을 명시적으로 들어냄 (유효하다)**
- 암묵적인 금액이라는 개념 
	- `long price`
	- `long totalPrice`
- Money 클래스로 로직 구현
	- 금액과 금액 계산 로직을 명시적인 Money 값 객체를 이용해서 표현
	- `Money price`
	- `Money totalPrice`
	- 금액의 개념을 명확하게 표현하게 되어 이해하기 쉬워짐 

> 값 객체는 중복 제거 위해 사용할 수 있다
- ex. 금액 올림 로직이 중복되어 있다. 이것을 Money로 이동 시킨다

>[!info] DRY (Don't Repeat Yourself), 반복하지 마라
>모든 지식 조각은 시스템 내에서 하나의 모호하지 않고 권위있는 표현을 가져야 한다.
- 개념을 값 객체 (VO)로 만들어서 중복을 줄일 수 있다는 거구나✅
	- 요구사항이 변경될때 함께 변경될때 같이 변경된다 == 중복 코드


값 객체는 합성(Composition) 관계로 연결된다
- 값 객체를 포함하는 객체의 생명 주기에 종속 
	- Sales, Game 등과 함께 Money 객체 생성되고 함께 제거된다 

값 객체 설계
- 일반적인 하향식 설계 방식이 아닌 리팩터링을 통해 식별
- 코드 리팩터링 중에 복잡하거나 중복된 로직을 간단한 개념을 값 객체로 추출

**✅ 값 객체를 위한 가이드**
> 방법1. 참조 객체를 리팩터링하다가 중복되는 개념을 발견하면 그것을 값객체로 뽑는다
> 방법2. 여러 장소에서 **함께 사용되는 변수**들이 있다면 값 객체로 추출할 수 있을지 고민하라


### 4-3. 값 객체를 이용해서 Game 개선하기
> 코드 레벨 설명 

1. Position 
```java
private int x, y; // 좌표값이 함께 중복 사용되고 있다. 
```
- `위치`라는 개념이 숨겨져 있다.
	- 값 객체를 만들때 변수뿐만 아니라 메서드도 같이 옮긴다
	- **Position 객체 생성**한다
	- ✅ 값 객체는 정적 팩터리 메서드를 사용하면 가독성 개선 가능

2. Size 
```java
private int width, height;
```
- 함께 사용되고 있다.
- Size 값 객체를 추가한다

> 📌 일단 값 객체를 만들어 놓으면 
> - 크기가 작기 때문에 여러 곳에서 재사용 가능하다
> - 복잡한 로직을 담을 수 있는 적합한 장소 제공 가능하다
- `"코드를 리팩터링할때 복잡한 로직이 보이면 값 객체로 옮길 수 없는지 고민한다"`고함


3. 방향 개념을 표현하는 Direction을 이용해서 명시적으로 표현
```java
public enum Direction {
	NORTH, SOUTH, EAST, WEST
}
```
- `position.shift(..)`에다가 적용


> [!tip] 복잡한 로직을 재사용 가능한 값 객체로 이동시켜 전체 설계를 단순화할 수 있다
- 읽기 어려운 코드가 있으면 아래 두 가지를 수행하신다 함
	- 1. 긴 메서드를 조합 메서드로 분리 
	- 2. 값 객체로 분리

**Q. 참조 객체와 값 객체의 가장 중요한 차이점 중 하나는 객체의 같음(Equality) 여부를 판단하는 기준입니다. 이 기준은 무엇일까요?**

A. 값 객체는 그 '값' 자체에 의미가 있어 내부 속성 값이 같으면 동일하다고 보지만, 참조 객체는 고유한 '식별자'가 중요하여 메모리 주소가 같아야 동일하다고 판단합니다.


**Q. 참조 객체에서 '별칭(Alias)' 사용 시 발생할 수 있는 주요 문제점은 무엇일까요?**

A. 별칭은 동일한 참조 객체를 가리키므로, 하나의 별칭을 통해 객체의 상태를 변경하면 같은 객체를 참조하는 다른 별칭들에서도 이 변화가 바로 반영되어 예상치 못한 부작용이 발생할 수 있어요.


**Q. 값 객체를 구현할 때 불변(Immutable) 객체로 만드는 것이 권장되는 주된 이유는 무엇일까요?**

A. 값 객체를 불변으로 만들면 상태 변경 시 항상 새로운 객체가 생성되기에, 객체의 상태 변화를 쉽게 추적할 수 있고, 별칭이 있더라도 상태 변경이 다른 별칭에 영향을 주지 않아 사이드 이펙트 발생 가능성을 줄여줍니다.


**Q. 값 객체가 코드의 전반적인 복잡성을 낮추는 데 기여하는 방식이 아닌 것은 무엇인가요?**

A. 값 객체는 개념 명시, 응집도 증가, 불변성을 통한 상태 관리 단순화로 복잡성을 줄여주지만, 메모리 사용량 감소가 주된 목적이나 효과는 아닙니다. 불변 객체 생성은 오히려 메모리 할당을 늘릴 수도 있어요.


**Q. 기존 클래스에서 값 객체로 분리하여 개선할 수 있는 부분을 찾아내는 일반적인 방법은 무엇일까요?**

A. 함께 사용되는 변수들은 종종 숨겨진 특정 개념(예: 위치, 크기)을 나타낼 수 있으며, 이 묶음을 값 객체로 분리하면 코드가 더 명확해지고 관리가 쉬워집니다. 리팩토링 시 이런 부분을 주목하세요.
