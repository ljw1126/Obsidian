AWS EC2나 Azure App Service처럼 앱을 배포하고 운영하는 클라우드 플랫폼입니다. 차이점은:

// Drizzle (TypeScript)

  .select()
  .from(ticketsTable)
  .where(eq(ticketsTable.status, 'TODO'))
  .orderBy(asc(ticketsTable.position));

// Spring JPA

차이점은 코드 생성이 없다는 것입니다. Hibernate는 @Entity 클래스를 보고 SQL을 생성하고, Prisma(JS의 다른 ORM)는 prisma generate로 타입을 뽑아냅니다. Drizzle은 스키마 파일 자체가 TypeScript 타입이라 별도 생성 단계가 없습니다.

// Zod (TypeScript)
const createTicketSchema = z.object({
  title: z.string().min(1).max(200),

// Spring Bean Validation
public class CreateTicketRequest {
  @NotBlank @Size(max = 200)
  private String title;

  @Size(max = 1000)
  private String description;

실무 팀                     이 방식
──────────────────────────────────────

실제 PRD는 "왜"에서 시작합니다.

▎ "이 앱을 왜 만드는가? 누가 쓰는가? 어떤 문제를 푸는가?"

지금 PRD는 기능 명세에 가깝고, 비즈니스 맥락(경쟁사, 수익 모델, 지표)이 없습니다. 실무 PM이 작성한 PRD는 아닙니다.

RSC란?

React Server Components — 서버에서만 실행되는 React 컴포넌트입니다.

Next.js App Router에서는 기본적으로 모든 컴포넌트가 RSC입니다.

기존 방식 (Pages Router)
  서버: HTML 생성 → 브라우저: React 전체 재실행 (hydration)

App Router (RSC)
  서버: 일부 컴포넌트를 완전히 서버에서 렌더링 → 브라우저: 클라이언트 컴포넌트만 hydration

Spring으로 비유하면:

RSC (서버 컴포넌트)   ≈  Thymeleaf 템플릿 (서버에서 HTML 생성, JS 없음)
클라이언트 컴포넌트   ≈  React SPA (브라우저에서 상태 관리, 이벤트 처리)

구분 방법: 파일 최상단에 'use client'가 있으면 클라이언트 컴포넌트, 없으면 RSC입니다.

// RSC — DB 직접 조회 가능, useState/useEffect 사용 불가
export default async function Page() {
  const tickets = await db.select().from(ticketsTable); // 서버에서 실행
  return <BoardContainer initialData={tickets} />;
}

// 클라이언트 컴포넌트
'use client';
export function BoardContainer({ initialData }) {
  const [board, setBoard] = useState(initialData); // 브라우저에서 실행
}


예제의 설계가 이 패턴입니다:

App (page.tsx)          ← RSC: DB에서 초기 데이터 fetch
└── BoardContainer      ← 'use client': 상태 관리, DnD, 이벤트

SDD(명세 주도 개발)가 맞습니다

다만 전통적 SDD와 차이가 있습니다.

전통적 SDD
사람이 명세 작성 → 사람이 구현 → 사람이 테스트

여기서 하는 방식
사람이 docs/ 작성 (What/Why)
    ↓
AI가 specs/ 생성 (How 설계)
    ↓
AI가 TDD 사이클 수행 (Red → Green → Refactor)
    ↓
사람이 검토 및 승인

핵심 원칙은 동일합니다. "명세가 구현보다 항상 앞선다" — 코드를 먼저 짜고 명세를 맞추는 게 아니라, 명세를 먼저 확정하고 코드를 맞춥니다. 실제로 이번 작업에서도 API_SPEC.md와 TEST_CASES.md가 충돌했을 때 코드가 아닌 명세를 기준으로 판단했습니다.

두 레이어의 역할 차이
┌───────────┬────────────────────┬─────────────────────────┐
│           │       docs/        │       specs/NNN/        │
├───────────┼────────────────────┼─────────────────────────┤
│ 작성 주체 │ 사람 (기획/설계)   │ AI (Speckit)            │
├───────────┼────────────────────┼─────────────────────────┤
│ 수명      │ 프로젝트 전체      │ 해당 기능 개발 기간     │
├───────────┼────────────────────┼─────────────────────────┤
│ 성격      │ 권위 문서 (Oracle) │ 실행 계획서 (Contract)  │
├───────────┼────────────────────┼─────────────────────────┤
│ 변경      │ 신중하게 (팀 합의) │ 자유롭게 (구현 과정 중) │
├───────────┼────────────────────┼─────────────────────────┤
│ 충돌 시   │ 우선               │ 따라감                  │
└───────────┴────────────────────┴─────────────────────────┘

docs/는 "무엇을 왜 만드는가" 의 기준점이고, specs/는 "어떻게 만들 것인가" 의 실행 지침입니다. Speckit은 그 사이를 자동화하는 도구입니다.

TC-INT-001/002는 어떤 테스트인가?

BoardContainer 수준에서 여러 컴포넌트가 실제로 협력하는 흐름을 검증하는 컴포넌트 통합 테스트입니다. API는 jest.mock으로 대체합니다:

[실제 BoardContainer]
  → [실제 Board + Column + TicketCard]
  → [실제 useTickets 훅]
  → [ticketApi 모킹] → jest.fn() 리턴

TC-INT-001 예시 (드래그앤드롭 + API 연동):
// @dnd-kit의 이벤트를 직접 fire해서 DnD 흐름 전체를 검증
fireEvent.pointerDown(card);
fireEvent.pointerMove(doneColumn);
fireEvent.pointerUp(doneColumn);

// complete API가 호출되었는지 확인
expect(mockCompleteTicket).toHaveBeenCalledWith(ticketId);

왜 아직 미작성인가?

dnd-kit의 포인터 이벤트를 jsdom에서 시뮬레이션하는 것이 상당히 까er.test.tsx에서 이미 boardProps.onDragStart/onDragEnd를 직접호출하는 방식으로 로직은 검증했지만, TC-INT-001/002는 컴포넌트를 실제로 마운트하고 DnD 이벤트까지 시뮬레이션하는 더 높은 수준의 테스트입니다.

진짜 E2E(실제 브라우저 기반)를 원한다면 Playwright나 Cypress를 별도로 붙여야 합니다. TEST_CASES.md의 TC-INT는 그 중간 단계인 컴포넌트 통합 수준으로 설계되어 있습니다.
