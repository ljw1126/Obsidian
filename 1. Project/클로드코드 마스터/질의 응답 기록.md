Vercel

AWS EC2나 Azure App Service처럼 앱을 배포하고 운영하는 클라우드 플랫폼입니다. 차이점은:

- Git push 하면 자동 배포 (CI/CD 설정 불필요)
- PR 올리면 Preview URL이 자동 생성됨 (https://tika-pr-42.vercel.app)
- Serverless로 동작 — 서버를 직접 관리하지 않음

Spring Boot를 직접 jar로 서버에 올리는 게 아니라, Vercel이 서버 관리를 대신 해주는 구조입니다.

Drizzle ORM

Spring의 JPA/Hibernate, .NET의 Entity Framework Core와 정확히 같은 역할입니다.

// Drizzle (TypeScript)
const tickets = await db
  .select()
  .from(ticketsTable)
  .where(eq(ticketsTable.status, 'TODO'))
  .orderBy(asc(ticketsTable.position));

// Spring JPA
List<Ticket> tickets = ticketRepository
  .findByStatusOrderByPositionAsc(Status.TODO);

차이점은 코드 생성이 없다는 것입니다. Hibernate는 @Entity 클래스를 보고 SQL을 생성하고, Prisma(JS의 다른 ORM)는 prisma generate로 타입을 뽑아냅니다. Drizzle은 스키마 파일 자체가 TypeScript 타입이라 별도 생성 단계가 없습니다.

Zod



Spring의 @Valid + Bean Validation(@NotNull, @Size, @Pattern), .NET의 Data Annotations + FluentValidation과 같은 역할입니다.

// Zod (TypeScript)
const createTicketSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(1000).nullable(),
  priority: z.enum(['LOW', 'MEDIUM', 'HIGH']).default('MEDIUM'),
});

// Spring Bean Validation
public class CreateTicketRequest {
  @NotBlank @Size(max = 200)
  private String title;

  @Size(max = 1000)
  private String description;
}

Zod의 핵심 강점은 프론트와 백엔드가 같은 스키마를 공유한다는 점입니다. Spring에서는 Request DTO를 백엔드에서만 검증하지만, Zod 스키마는 src/shared/에 두고 클라이언트(폼 검증)와 서버(API 검증) 양쪽에서 import해서 씁니다.

공통점 (JPA와 같은 것)

- SQL을 직접 쓰지 않고 코드로 DB 쿼리 작성
- 테이블 ↔ 코드 객체 매핑
- 마이그레이션 관리 (스키마 변경 추적)

---
차이점

┌─────────────┬────────────────────────────────────┬──────────────────────────────────────────┐
│             │          JPA / Hibernate           │               Drizzle ORM                │
├─────────────┼────────────────────────────────────┼──────────────────────────────────────────┤
│ 패러다임    │ ORM — 객체가 주체, SQL은 자동 생성 │ Query Builder — SQL 구조를 코드로 표현   │
├─────────────┼────────────────────────────────────┼──────────────────────────────────────────┤
│ 매핑 방식   │ @Entity 클래스 → 테이블 자동 매핑  │ 스키마 파일을 직접 정의 → 타입 자동 추론 │
├─────────────┼────────────────────────────────────┼──────────────────────────────────────────┤
│ 코드 생성   │ persistence.xml / 어노테이션 기반  │ 없음 (스키마 파일 자체가 타입)           │
├─────────────┼────────────────────────────────────┼──────────────────────────────────────────┤
│ 추상화 수준 │ 높음 (SQL 거의 안 봄)              │ 낮음 (SQL과 1:1 대응, 예측 가능)         │
├─────────────┼────────────────────────────────────┼──────────────────────────────────────────┤
│ N+1 문제    │ 발생하기 쉬움                      │ 구조상 발생하기 어려움                   │
└─────────────┴────────────────────────────────────┴──────────────────────────────────────────┘

// JPA — 객체 중심, SQL은 Hibernate가 알아서
@Entity
@Table(name = "tickets")
public class Ticket {
    @Id @GeneratedValue
    private Long id;

    @Column(nullable = false, length = 200)
    private String title;
}

// 쿼리
ticketRepository.findByStatusOrderByPositionAsc("TODO");

// Drizzle — SQL 구조를 코드로 그대로 표현
const tickets = pgTable('tickets', {
  id:     serial('id').primaryKey(),
  title:  varchar('title', { length: 200 }).notNull(),
  status: varchar('status').notNull(),
});

// 쿼리 — SELECT * FROM tickets WHERE status='TODO' ORDER BY position ASC 와 1:1
const result = await db
  .select()
  .from(tickets)
  .where(eq(tickets.status, 'TODO'))
  .orderBy(asc(tickets.position));

한 줄 요약

JPA는 "객체를 저장한다" 는 추상화, Drizzle은 "SQL을 타입 안전하게 쓴다" 는 추상화입니다.

JS 진영에서 JPA에 더 가까운 건 Prisma입니다. Drizzle은 의도적으로 SQL에 가깝게 설계된 라이브러리입니다. Tika가 Drizzle을 선택한 이유도 TRD에 적혀있듯 "코드 생성 없음 + Vercel Postgres 공식 지원" 때문입니다.
