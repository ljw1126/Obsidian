## 3 Chpater. AI와 함께하는 개발 방법론 

> [!info] AI 시대의 역설 - 바이브 코딩 숙취
- 코드 생성이 더 쉬워졌기 때문에 오히려 유지보수와 기능의 완성도를 높이기 위한 설계가 무엇보다 더 중요했다.
- "일단 만들어보고 아니면 다시 만들면 되지"라는 생각의 접근 (함정💩)
	- AI는 코드를 빠르게 만들지만, 그 코드가 올바른 방향인지는 알려주지 않고, 보장해주지도 않는다.
	- 잘못된 방향으로 코드가 나오기 시작하면 무한 수정과 유지보수 지옥에 빠질 가능성이 높다. 
	- 소프트웨어의 생명력은 초기엔 아이템으로 승부가 나지만 이 시기가 지나면 사용자에게 유용한 기능을 추가하고, 기존 기능들을 원활하게 유지하며 업데이트를 하는 운영의 영역에서 판가름 난다고 해도 과언이 아니다. 
	- 장애나 버그가 잦으면 사용자는 떠나기 마련이다. 

p145 설계의 정의와 범위 
- AI와 협업하는 맥락에서 설계는 그보다 훨씬 실용적인 의미를 갖는다. 
	- **요구사항 분석** : 무엇을 만들 것인가? 핵심 기능과 비기능적 요구사항을 수립한다. 
	- **아키텍처 결정** : 어떤 기술과 구조로 만들 것인가? 구현부터 빌드/배포에 이르기까지 영향을 미치므로 클로드 코드나 제미나이와 같은 에이전트의 도움을 받아서 결정하는 것도 좋은 방법이다.
	- **인터페이스 정의** : API 호출 방식이나 규격 등을 정의하여 데이터를 어떤 식으로 보여줄지, 시스템 간 호출은 어떻게 정의할지를 미리 작성해야 한다.

p241
**4.5.1 SDD란 무엇인가**

> TDD는 `'코드가 의도대로 동작하는가'`를 테스트로 검증하는 **품질 보증 방법론**이고, SDD는 `'무엇을 만들어야 하는가'`를 명세로 선언하는 **개발 프로세스 방법론**이다. TDD의 산출물은 테스트 코드이고, SDD의 산출물은 명세 문서다. 경쟁 관계가 아니라 레이어가 다르다.

> SDD가 'What'을 정의하면, TDD가 'How'를 검증


p244 SDD흐름 (5단계)
- 1. 요구사항 분석
- 2. 명세 문서 작성
- 3. 명세 리뷰 및 확정
- 4. 명세 기반 구현
- 5. 명세 대비 검증 


SDD의 핵심 가치는 다음과 같다 
- **바이브 코딩 방지** : "대충 이렇게 해줘"가 아닌 명확한 명세 기반 개발 
- **맥락 유지** : 대화가 길어져도 AI가 프로젝트의 목표와 제약을 잊지 않음 
- **협업 표준화** : 팀원 누구나 AI를 사용해도 동일한 품질의 결과물
- **의도 중심 개발** : 코드가 아닌 `의도(intend)`가 중심이 되는 개발

drizzle.config.ts — drizzle-kit CLI 전용 (개발 도구)

// 이 파일은 "npm run db:generate", "npm run db:migrate" 실행 시에만 읽힌다

  dbCredentials: { url: process.env.DATABASE_URL! }, // 어느 DB에 마이그레이션 적용할지
});

npx drizzle-kit generate → schema.ts를 읽어 SQL 파일 생성
npx drizzle-kit migrate → 생성된 SQL을 DB에 적용

const client = postgres(process.env.DATABASE_URL!);
