```shell
   * 대규모 프로젝트: 시스템이 커지면 하위 모듈(인프라) 코드가 너무 많아져서, 어쩔 수 없이 infrastructure 패키지로 밀어내는 방식을 취하게 됩니다. 하지만 이때도 인터페이스만큼은
```

```shell
  1. 아이템 소실 버그 (Item Loss Bug)

  현재 World.throwItem()을 실행하면 Transfer 클래스를 통해 아이템 이동이 일어납니다.

   2. 아이템 제거: Transfer는 먼저 플레이어(Source)의 인벤토리에서 아이템을 제거합니다.
   3. 아이템 추가: 그 후 worldMap.add(item)을 호출합니다.

  기존 WorldMap.add 코드를 보면:

   1 public void add(Item item) {
   2     Position position = size.anyPosition(); // 무작위 위치 선택
   3     if(isBlocked(position)) {
   4         return; // 버그 발생 지점: 위치가 막혀있으면 그냥 종료
   5     }
   6     roomAt(position).add(item);
   7 }
  여기서 isBlocked(position)이 true가 되면(즉, 해당 위치에 방이 없으면), 아이템을 어디에도 추가하지 않고 메서드가 종료됩니다. 결과적으로 아이템은 플레이어
  인벤토리에서 이미 지워졌는데, 월드맵 어디에도 생성되지 않아 영원히 소실되는 버그가 발생합니다.

  4. 'Tell, Don't Ask' 원칙 위반

  현재 코드는 WorldMap이 isBlocked(position)을 호출하여 상태를 물어보고(Ask), 결과에 따라 로직을 결정하고 있습니다.

   * 문제: 외부에서 isBlocked라는 상태를 직접 확인하고 판단하는 로직이 흩어져 있습니다.
   * 개선 방향: 특정 위치(Location)에게 "여기에 아이템을 추가해줘"라고 명령(Tell)하고, 그 위치가 아이템을 수용할 수 있는지 스스로 판단하게 해야 합니다.

  ---

  개선 전략

   1. 소실 방지: 아이템이 반드시 배치될 때까지 유효한 위치를 찾도록 루프를 수행합니다.
   2. 객체 지향적 캡슐화: Location이라는 개념을 도입하여, 해당 위치가 "차단된 곳인지", "아이템 추가가 가능한지"를 스스로 처리하도록 합니다.

  이렇게 하면 WorldMap은 "무작위 위치의 Location에게 아이템 추가를 부탁"하는 형태가 되어 코드가 더 간결하고 안전해집니다.

✦ 사용자님께서 처음에 주신 프롬프트에 "아이템이 소실되는 버그가 있습니다"라고 명시해주셨기 때문에, 저는 이를 해결해야 할 기술적 결함으로 판단했습니다. 하지만
  기획적인 관점에서는 사용자님의 말씀대로 "벽 너머나 허공으로 던져진 아이템은 사라진다"는 것이 의도된 동작일 수도 있습니다.

  그럼에도 불구하고 개발자 입장에서 이것을 '버그' 혹은 '나쁜 코드'로 보는 이유는 다음과 같습니다.

  1. 전이(Transfer)의 원자성 결여
  현재 World.throwItem() 로직은 Transfer 객체를 사용합니다.
   * 성공 메시지: 아이템이 인벤토리에서 제거되면 "어딘가로 던졌습니다"라는 메시지가 나옵니다.
   * 실제 결과: 하지만 WorldMap.add() 내부에서 isBlocked에 걸리면 아이템은 그냥 증발합니다.
   * 문제점: 사용자는 성공했다는 피드백을 받았는데, 실제 월드에는 아이템이 존재하지 않는 상태 불일치가 발생합니다. 만약 소실이 의도된 것이라면 "아이템이 허공으로
     사라졌습니다"와 같은 명확한 피드백이 있거나, 처음부터 Destroy와 같은 별도의 개념으로 처리되는 것이 안전합니다.

  2. 'Tell, Don't Ask' 원칙 위반 (더 중요한 이유)
  이 버그가 발생한 근본 원인은 WorldMap이 Position에게 상태를 캐물어서(Ask) 스스로 판단하기 때문입니다.

   * 현재 방식 (Ask): "너 막혀있니? 아니라고? 그럼 거기 있는 방을 줘봐. 이제 그 방에 아이템을 넣을게."
       * 이 과정에서 isBlocked 체크와 roomAt 호출 사이에 로직이 끊기거나, 체크를 강제하지 못해 소실 같은 실수가 발생하기 쉽습니다.
   * 지향하는 방식 (Tell): "이 위치(Location)에 아이템을 추가해줘."
       * WorldMap은 위치 객체에게 아이템을 던질 뿐이고, 추가 성공 여부나 위치의 유효성은 그 위치 객체가 스스로 판단하게 합니다.

  요약
  사용자님의 말씀대로 "반드시 던지기에 성공해야 한다"는 요구사항이 없다면 루프(while)를 돌 필요는 없습니다. 하지만 'Tell, Don't Ask'를 적용한다면, 최소한 외부에서
  isBlocked를 체크하고 조건문을 실행하는 구조는 개선하는 것이 좋습니다.

  질문:
  아이템 소실이 게임의 의도(확률적 소실 등)라면 루프를 빼고 구조만 개선할까요? 아니면 "던지면 반드시 어딘가(방)에는 떨어져야 한다"는 전제하에 루프를 포함하여
  수정할까요?
```

```shell
 > 최근에 Location 과 같이 inner class 형태로 선언해서 그 클래스 내부에서만 활용하기도 합니다. 이러한 방식이 좋은 습관일까요?

✦ 네, 결론부터 말씀드리면 매우 좋은 습관이며 권장되는 객체지향 설계 기법 중 하나입니다.

  이러한 방식(Inner Class/Record를 활용한 내부 추상화)이 왜 좋은 습관인지 몇 가지 핵심적인 이유를 정리해 드립니다.

  1. 문맥(Context)의 캡슐화
  Location이라는 개념이 WorldMap 밖에서도 의미가 있다면 독립적인 클래스로 만들어야 하겠지만, 현재처럼 "월드맵의 특정 지점에 대한 일시적인 상태와 행위"를 다루는
  용도라면 내부 클래스가 가장 적절합니다.
   * 외부에서는 WorldMap만 알면 되고, 그 안에서 Location이 어떻게 동작하는지(정보 은닉) 알 필요가 없습니다.

  2. 'Tell, Don't Ask' 구현의 징검다리
  'Tell, Don't Ask'를 지키려고 할 때 가장 흔히 범하는 실수가 외부에 있는 객체(예: Room)를 가져와서 복잡한 로직을 수행하는 것입니다.
   * 기존: worldMap.roomAt(pos)로 객체를 꺼낸 뒤, null 체크를 하고, 다시 add를 호출.
   * 개선: at(pos)라는 메서드로 Location이라는 행위의 주체를 만들고, 그 주체에게 명령(add)을 내림.
   * 이때 Location은 복잡한 로직을 감싸주는 일종의 어댑터나 래퍼(Wrapper) 역할을 하여 코드를 훨씬 읽기 좋게 만듭니다.

  3. 코드의 가독성과 의도 명확화 (Internal DSL)
  내부 클래스를 사용하면 코드가 서술적으로 변합니다.

   1 // 무엇을 하는지 한눈에 들어옵니다.
   2 at(position).add(item);
  단순히 Room 객체를 직접 다루는 것보다, Location이라는 개념을 통해 "특정 위치에 대한 처리를 한다"는 개발자의 의도가 더 명확하게 전달됩니다.

  4. 낮은 결합도 (Low Coupling)
  만약 Location을 별도 파일로 만들면, 다른 클래스들도 Location에 의존하게 될 가능성이 생깁니다. 하지만 WorldMap 내부의 private 또는 package-private record로 두면, 이
  로직은 오직 WorldMap의 통제 하에 있게 되어 전체 시스템의 결합도가 낮아집니다.

  ---

  ⚠️ 주의할 점 (언제 분리해야 할까?)
  다만, 무조건 좋은 것은 아니며 아래와 같은 상황에서는 독립적인 클래스로 추출(Extract Class)하는 것이 좋습니다.

   1. 재사용성: Player나 Monster 클래스에서도 Location 객체의 기능이 똑같이 필요할 때.
   2. 비대해지는 클래스: WorldMap 코드가 너무 길어져서 내부 클래스 로직 때문에 전체적인 가독성이 오히려 떨어질 때.
   3. 독립적 테스트: Location의 로직이 매우 복잡해서 WorldMap을 통하지 않고 단독으로 유닛 테스트를 정밀하게 수행해야 할 때.

  요약하자면:
  지금처럼 특정 클래스 내부의 복잡한 로직을 정돈하고 'Tell, Don't Ask'를 실현하기 위한 도구로 사용하는 것은 실무에서도 자주 쓰이는 아주 세련된 방식입니다. 특히 Java
  16부터 도입된 record를 활용하면 보일러플레이트 코드 없이 깔끔하게 구현할 수 있어 더욱 권장됩니다.
```
