
### 3-1. 단일 추상화 수준 원칙과 조합 메서드
> 메서드를 작게 만들어야 하는 이유에 대해 설명

**긴 메서드를 작은 메서드로 나누기**
- 작은 메서드는 긴 메서드에 비해 클래스를 옮기기도 쉽고 설계를 리팩터링하기 수월해진다

**이해하기 어려운 긴 메서드가 있을때**
- 독립적인 메서드로 추출한 후 `의도`를 드러내는 `이름`을 붙여라

>[!info] 메서드 추출에서 가장 중요한 것
>✅ 의도를 드러내는 이름 (Intention Revealing Name)
>- 메서드의 이름은 메서드 **호출자의 의도**를 반영하도록 명명
>- 메서드 호출자의 목표(what)를 설명하고 이를 달성하는 매커니즘이나 알고리즘(how)을 드러내지 말것
- 메서드 이름은 : 어떻게(how) 구현하는지가 아니라 무엇을(what) 하는지

```java
// 👎
Array::linearSearch
Set::hashedSeard
TreeSet::binarySearch

// 👍 , 클라이언트 의도를 표현한다
Collection::find
```


>[!info] 마틴 파울러, 리팩터링 
>나는 길이가 중요하다고 생각하지 않는다. 중요한 것은 **메서드의 이름과 메서드 몸체 사이의 의미적 차이**다. 뽑아내는 코드를 더욱 명확하게 하면, 새로 만든 메서드의 이름이 원래 코드의 길이보다 길어져도 뽑아낸다.
- 메서드 구현과 이름 사이의 의미적 차이가 중요 


**📌메서드 추출의 이유**
- ✅**가독성 향상(제일 중요)** : 로직을 읽기 쉽게 개선하기 
- **중복 코드 제거** : 중복 코드를 공통 메서드로 추출하기
	- 메서드 추출하고, 메서드로 대체
- **불안정한 의존성 고립** : 의존성을 메서드 내부로 제한하기
	- 제어하기 어렵고 테스트하기 어려운 의존성을 메서드 추출로 분리하면 코드의 유지보수성을 높일 수 있다.


**읽기 쉬운 메서드의 핵심**
> 메서드 안의 모든 코드가 동일한 추상화 수준에 있도록 메서드 추출


**SLAP (Single Level of Abstraction Principle, 단일 추상화 수준 원칙)**
>[!info] 메서드 내의 모든 코드는 동일한 추상화 수준에 위치해야 합니다. 
>즉, 메서드의 일부가 저수준의 데이터베이스에 연결을 처리하고, 다른 부분은 고수준의 비즈니스 코드를 처리하며, 또 다른 부분은 웹 서비스 관련 코드를 처리하는 일이 없어야 합니다. 물론, 이런 메서드는 켄트 벡의 조합 메서드 (Composed Method) 규칙도 위반하게 됩니다.  - 닐 포드, 능률적인 프로그래머

> [!info] ✅ 의도를 드러내는 이름 (Intention Revealing Name)
>- 메서드의 이름은 메서드 **호출자의 의도**를 반영하도록 명명
>- <u>메서드 호출자의 목표(what, 고수준)</u>를 설명하고 <u>이를 달성하는 매커니즘이나 알고리즘(how, 저수준)</u>을 드러내지 말것

>[!warning] 추상화 수준이 섞여 있다면?
>이해하기 어려운 메서드가 만들어진다. (코드 예시 설명)
- 추상화 수준이 다르면 다시 원래의 메서드로 돌아와서 로직을 이해하여야 한다
	- 서로 다른 수준의 정보를 조합해야 하는 어려움 발생 (결국 이해 어려워짐)

> 추상화 수준이 동일하면 왔다갔다 할 필요없이 어떤 일을 하는지 쉽게 알 수 있다. (메서드의 목적을 알 수 있다. 세부적인거는 궁금하면 메서드로 들어가면 된다)

```java
// 추상화 수준이 동일하다
public void run() {
	welcome();
	play();
	farewell();
}
```
- 이런걸 **조합 메서드(Composed Method)**

> [!info] 켄트벡, 스몰토크 모범사례 패턴
> 프로그램을 하나의 명확한 작업을 수행하는 메서드들로 나누세요.
> 메서드 내의 모든 작업을 **동일한 추상화 수준**으로 유지하세요.
> 이렇게 하면 여러 개의 작은 메서드로 이루어진 프로그램이 만들어지고, 각 메서드는 몇줄 정도로 짧아지게 됩니다.
- 무슨 일을 하는지 이해하기 어려운 코드가 있다면 
	- 추상화 수준에 따라 코드를 나누고
	- 분류된 작업을 의도를 드러내는 메서드로 추출
	- 추출한 메서드를 조합한 새로운 메서드를 이용해서 가독성 향상 
- 단일 추상화 수준 원칙을 준수할 수 있다




### 3-2. 조합 메서드로 리팩터링 하기
**SLAP (Single Level of Abstraction Principle, 단일 추상화 수준 원칙)**

>[!info] 메서드 내의 모든 코드는 동일한 추상화 수준에 위치해야 합니다. 
>즉, 메서드의 일부가 저수준의 데이터베이스에 연결을 처리하고, 다른 부분은 고수준의 비즈니스 코드를 처리하며, 또 다른 부분은 웹 서비스 관련 코드를 처리하는 일이 없어야 합니다. 물론, 이런 메서드는 켄트 벡의 조합 메서드 (Composed Method) 규칙도 위반하게 됩니다.  - 닐 포드, 능률적인 프로그래머


**동일한 추상화 수준으로 나누기** 
> 이번 시간에 Game.run()을 리팩터링 해본다 
- 비슷한 메서드를 묶고 주석을 단다. 
- 실행 단계를 동일한 수준으로 추상화 
	- 이제 메서드를 추출한다 
		- 좋은 이름 : 클라이언트의 의도를 표현하는 이름
			- run()이 클라이언트 코드다

```java
// 조합 메서드 (Composed Method)
public void run() { // 추상화 수준이 동일한 코드들을 조합해서 구현
	welcome();
	play();
	farewell();
}
```

>[!info] 프로그램을 하나의 명확한 작업을 수행하는 메서드들로 나누세요. 
>메서드 내의 모든 작업을 동일한 **추상화 수준**으로 유지하세요. 이렇게 하면 여러 개의 작은 메서드로 이루어진 프로그램이 만들어지고, 각 메서드는 몇 줄 정도로 짧아지게 됩니다. - 켄트백, 스몰토크 모범 사례 패턴


> welcome()을 한번더 조합 메서드로 만들 수 있다. (동일한 추상화 수준으로 나누기)


동일한 추상화 수준으로 메서드를 조합하면 **이해하기 쉬운 코드**가 만들어진다
```java
// 게임을 실행run하기 위해서는
public void run() {  
	welcome(); // 환영 메시지를 출력하고
	play(); // 게임을 플레이한 후
	farewell(); // 작별 메시지를 출력한다.
}
```

> 마치 문장처럼 읽을 수 잇다.

```java
// 환영 메시지를 출력하기 위해서는
private void welcome() {
	showGreetings(); // 인사말을 표시 하고
	showRoom(); // 현재 방의 정보를 표시하고
	showHelp(); // 도움말을 표시한다
}
```


>[!info] 언제까지 메서드를 추출할 것인가?
>메서드가 **오직 한 가지 작업만 수행**하게 될 때까지 추출하라
>= 메서드가 **오직 한 가지 이유로 변경**될 때까지 추출하라
- 한가지 작업/이유 = 클라이언트가 달성하려는 한 가지 목적을 의미 

**✅ 메서드 수준의 응집도 (Cohesion)**
- 메서드가 여러 가지 이유로 변경이 된다면 메서드의 응집도가 낮다고 말한다
	- `낮은 응집도 (Low Cohesion)`
	- 반대로 메서드가 한 가지 이유로 변경된다면 응집도가 높다고 말한다
- ex. 리팩터링 전의 `run()`이 응집도가 낮았다
 

하지만 여전히 복잡한 play 메서드 .. 

**작은 메서드로 나눠서 코드를 읽기 쉽게 개선하기** 
- 가독성 향상 
	- 로직을 읽기 쉽게 개선하기 
- 중복 코드 제거 
	- 중복 코드를 메서드로 추출하기 
- 불안전한 의존성 
	- 의존성을 메서드 내부로 제한하기 

> play 메서드안에 이동 메서드가 복잡하다 .. **메서드로 추출한다**


```java
// before
// 의도를 나타내기 어렵다
this.running = true;
this.running = false;

// after
// start()
private void start() {
	this.running = true;
}

// isRunning()
private void isRunning() {
	return this.running == true;
}

// stop()
private void stop() {
	this.running = false;
}
```
- 의도가 들어나도록 메서드 추출 

동서남북 -> showRoom()
roomAt 메서드 추출
showBlocked() 메서드 추출 
tryMove() 메서드 추출 
- isBlocked 메서드 추출
	- 추상화 수준이 일정하지 않음 
- isExclued 메서드 추출해서 추상화 수준 동일하게 함 // 조건문도 그렇구나 


> [!info] 한 클래스에서 다른 클래스로 메서드를 이동시키는 작업은  객체지향 용어로 표현하면 `책임의 이동 (Shift or Responsibility)`이라고 말한다.
- 클래스를 분리하는 두 가지 방법 
	- 1. 값 객체 
	- 2. 단일 책임 원칙
