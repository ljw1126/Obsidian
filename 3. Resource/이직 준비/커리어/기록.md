AzureFrameWork (2026년 3월 첫주)

**기술적 부채 및 안정성 문제**
- 1. 캐시 관리 부재 
	- Cosmos DB 호출 공통 모듈에서 메모리 임계치 도달시 자동 축출(Eviction)이나 갱신 정책 없이 **'서버 재시작 하면 된다'를 해결책으로 제시**함 (백엔드 팀장)
	- 이는 가용성이 핵심인 서비스에서 **치명적인 운영 리스크**임
		- ex. LoggerData 
- 2. 테스트 자동화 부재, 검증 체계 누락
	- 테스트 프레임워크(xUnit 등) 없이 **웹 프레임워크를 직접 호출하고 주석으로 처리하는 방식은 "회귀 테스트(Regression Test)"를 불가능하게 만듦**.
	- 테스트 프레임워크 없이 수동으로 코드를 호출하고 주석 처리하는 방식이 고착화되어 있습니다. 이는 **회귀 테스트가 불가능**함을 의미하며, 신규 기능 추가 시 기존 기능이 망가지는 '도미노 장애'를 막을 수 없습니다.
- 3. AI 의존적 개발의 한계 
	- 설계와 예외 처리에 대한 엔지니어링적 검토 없이 AI 생성 코드에만 의존하여 구현 속도만 강조하고 있습니다. 이는 겉으로는 빨라 보이나, 실제 운영 환경에서의 **안정성과 확장성을 담보할 수 없습니다.**

팀장 및 팀원의 이상 행동 
- LoggerData 정책 
	- 프로젝트 내부 로직에 대한 설명은 없고, 의미없는 말만 반복 (병건님, 팀장님)
		- "Logger(로거)는 10분 통계를 구하고, 10분 통계로 1시간 통계를 구합니다. 그리고 1시간 통계로 일일 통계를 구합니다"
	- 1차 코드 리뷰 전 
		- 데이터 카운트 누적합 버그
			- 팀장 : "모르는게 있으면 질문 하세요"라고 했지만, Teams에 팀 채팅방에 질문하니 침묵으로 회피 (스크린샷 첨부)
		- 타임존 버그 
			- 수집된 데이터는 UTC 타임존을 사용 중인데 로컬 테스트를 하는 경우 KST 타임존으로 잘못 계산되어 데이터가 틀리는 버그 발생 
			- 이에 대해 유지보수 담당자에게 로컬 테스트는 해봤냐고 질문했고, "예.. 해봤어요..(병건)"라고 답변 받음 
				- 로컬 테스트 의미 
					- (제 생각) 자기 컴퓨터에 인프라 환경 구축후 하는 테스트 
					- (병건님) 운영 서버에 프로젝트 배포 후 하는 테스트 
			- 운영 서버는 UTC 타임존이라 정상 동작하지만, 
			- 고로, 로컬 테스트를 해봤냐는 것은 이들에게 운영서버에서 프로젝트 배포 후 테스트를 하는 방식을 뜻한다고 판단함
	- 1차 코드 리뷰 당시 팀장 왈
		- "서비스, 레파지토리 패턴 적용하는거 말고 없죠?"
		- "누구 보다 Logger에 대해 잘 알아야 하고, 누구보다 Logger에 대해 잘 설명할 수 있어야 합니다."
	- 2차 코드 리뷰 받기 몇일 전
		- VOC 발생하여 기획팀원 두명이 찾아와 대화를 하는 것을 들음
			- "백엔드팀에 Logger 내부 로직 설명할 수 있는 사람이 한명도 없다"
			- 용한님이 제 눈치를 살핌
	- 2차 코드 리뷰 후
		- 리팩터링 결과로 하드코딩된 1500줄이 100줄이 된 결과를 보고 아무 말이 없었음 (이상할 정도로 조용했었음)
		- 피드백을 Teams로 준다고 했는데 재입사한 시니어의 질문을 그대로 복붙한 답변이라 영양가가 없었음 
		- 유일하게 재입사한 시니어가 인터페이스 설계에 대한 의미있는 질문을 했었고 ISP나 캡슐화 원칙에 의거하여 유지보수 관점에서 답변 했었음
- AisData 업무 할당 후
	- `"저도 몰라요"` 
	- 기획 담당자도 이력을 모르고 팀장도 모르고, 시현님한테 물었을때 프로젝트 이름만 알고 있었음 
	- 코드 리뷰(발표) 시간에 설계에 대한 피드백을 받고 싶었지만 전혀 다른 질문을 하면서 지적 우월감을 어필
		- object vs dynamic 차이 아느냐 하면서 AI한테 질문을 하고 그걸 그대로 읇는 느낌을 받음 
	- "저는 dynamic 타입을 진짜로 혐오합니다"라고 말을 했지만 
		- 프로젝트 검색시 5400개 키워드 확인됨 
		- 말의 앞뒤가 맞지 않다고 생각함
- Leg Performance VOC
	- 인수인계 받은게 없는데 제 책임으로 전가 (Teams 대화 내용 첨부)
	- 그래서 VOC 발생 후 코드 레벨부터 분석하여 Jira에 댓글을 담 
		- 내가 분석한 원인에 대해 팀장이 댓글을 보고 
			- `"틀렸어요 (Teams)"`라고 말한 후 GOOD_WEATHER 조건에 대한 
	- VOC가 발생한 원인이 정책 습득 부재로 인한 관리팀의 문제였는데 
		- 이러한 문제를 알고 있다는 듯 "또 그거냐며?ㅋㅋ" 사람들이 웃는 모습을 확인할 수 있었음
